"코드(C++ core + Shell COM)는 그대로, UI 도구만 WinUI 3로 교체" 시나리오에서 성능이 어느 수준으로 맞춰질지를 검토한 보고서입니다. 4명의 에이전트가 베이스라인 추출 · 프레임워크 리서치 · C++/WinRT 시나리오 · C# 시나리오를 병렬로 조사하고, 마지막 종합 섹션에서 비교했습니다.

대상 커밋 92b999f · 베이스라인 ui-baseline.html 참조.

A.1 측정 인프라 개요

  • PerfTracker (src/core/perf-tracker.h:19) — 잠금-free MPSC 링 버퍼. 용량 kCapacity = 10'000 이벤트 (perf-tracker.h:21), 슬롯 = 24 B (perf-tracker.h:41 static_assert(sizeof(Event) == 24, ...)). 타임스탬프는 QueryPerformanceCounter로 캡처 (perf-tracker.cpp:40). 6개의 phase boundary 이벤트만 정의: AppLaunchStart / AppInteractive / AppShutdownStart / PaneOpenStart / PaneFirstBatch / MemoryProbe (perf-tracker.h:23-32). dump는 종료 시 OutputDebugString 또는 callback sink로 1회 수행 (perf-tracker.cpp:76-133). minidump의 user-stream에 raw slot buffer 첨부 가능 (perf-tracker.h:71-72).
  • RingLogger (src/core/ring-logger.h:27) — MPSC 링, kSlotCount = 512 · kSlotBytes = 256 · kMaxMessageBytes = 232 (= 256-24, ring-logger.h:38-40). 백그라운드 writer 스레드가 kFlushIntervalMs = 100 ms 마다 또는 INFO+ 즉시 flush (ring-logger.cpp:15, 189-190, 254). 백프레셔 시 메시지는 fallback(ODS) + dropped 카운터 증가 (ring-logger.h:22-26, 57-59).
  • StallProbe (src/ui/stall-probe.h:9-53) — UI 스레드 단일. classifyStall가 dispatch 갭을 kStallInfoMicros=50'000 / kStallWarnMicros=100'000 / kStallErrorMicros=500'000 μs 임계와 비교 (stall-probe.h:16-18). 7-bucket 히스토그램 (§A.3 표 참조). 메시지 루프의 Translate/Dispatch 양쪽에 QPC bracket으로 통합 (src/app/main.cpp:176-189). Error 등급에서는 PerfTracker dump를 로거로 ejection (main.cpp:152-154).
  • DispInfoHistogram (src/ui/dispinfo-histogram.h:22-53) — UI 스레드 단일. 7-bucket 마이크로초 히스토그램. handleGetDispInfo가 QPC bracket으로 every call 샘플 (main-window.cpp:2968-2983). p99 추정기는 ceil(0.99 × total) rank 기반 bucket-upper 보고 (dispinfo-histogram.cpp:39-59).
  • ProcessMemoryService (src/core/process-memory.h:35) — SetProcessWorkingSetSizeEx hint (kMinWorkingSet=8 MB / kMaxWorkingSet=128 MB, process-memory.cpp:17-18, 53-58, 두 hint 모두 QUOTA_LIMITS_HARDWS_*_DISABLE로 advisory). LowMemoryResourceNotification 백그라운드 thread + 5 s re-arm (kLowMemoryReArmIntervalMs=5000, process-memory.cpp:20, 217).
  • 덤프 경로: %LOCALAPPDATA%\FastExplorer\logs\fast-explorer-YYYYMMDD.log (RingLogger 헤더 ring-logger.h:11) 또는 portable override. 종료 시 main 루프에서 PerfTracker → StallHistogram → DispInfoHistogram 순으로 logger sink로 dump (src/app/main.cpp:386-390, 408).

A.5 디렉터리 열거 성능

  • batch flush 임계: Config::batchSize = 256 기본값 (directory-enumerator.h:30); batch.size() >= config_.batchSize 도달 시 flushBatch 호출, 종료/취소/오류 시에도 flush (directory-enumerator.cpp:94, 129-131, 98, 103, 122). 이 256 값이 design §14.3의 The DirectoryEnumerator batch boundary at 256 entries is what bounds this number 진술의 근거 (design.md:1400).
  • sharing violation 재시도: 기본 sharingViolationRetryMs = 100 / sharingViolationRetries = 1 (총 최대 2회 시도) (directory-enumerator.h:31-32). 재시도는 interruptible sleep으로 kPollingQuantumMs = 10 ms 마다 stop_token poll (directory-enumerator.cpp:37, 39-51).
  • FileModelStore 하드 한도: kMaxEntries = 100'000 (file-model-store.h:49); 이 값이 100k 데이터셋 워킹셋/소액 게이트의 기준선 (report.md:206-207, 388-389). FileEntry는 정확히 40 B (design.md:153, 399, file-entry static_assert).

A.9 베이스라인 요약 (다른 에이전트가 참조할 핵심 숫자)

  • GETDISPINFO budget: ≤ 50 µs / row (design.md:149, 342); histogram bucket boundary 일치 (1/5/20/50/100/500 µs).
  • UI stall classification: 50 / 100 / 500 ms = Info / Warn / Error (stall-probe.h:16-18); 종료 dump의 "≥500 ms bucket = 0"이 1-hour soak 패스 기준.
  • Frame budget: 60 Hz 기준 scroll p95 ≤ 16.7 ms (design.md:148).
  • Enumeration batch flush: 256 entries / sharing-violation 100 ms 재시도 × 1 / kMaxEntries 100 000 (directory-enumerator.h:30-32, file-model-store.h:49).
  • First batch latency (실측): small 4.05 ms / medium 3.62 ms / 100k 29.83 ms — 모두 game gate(50/100/200 ms) 통과.
  • 100k working set (헤드리스): baseline 4.04 MB → peak 9.39 MB → final 4.46 MB; 50 MB target의 18.8% (design.md:1520).
  • 10-cycle memory drift: 404 KB (5 MB 게이트의 8.1%; 1번째 사이클 이후 plateau, 누적 없음) (design.md:1521).
  • Sort (10k Name asc): 2.75 ms; 50 ms 게이트의 5.5% (design.md:1448).
  • ImageList footprint: ExtensionIconCache 256 슬롯 + placeholder 2 → ≈ 258–272 KB (3 MB cap의 ≤ 10%) (design.md:1478).
  • 측정 인프라 capacity: PerfTracker 10 000 events · 24 B/event; RingLogger 512 슬롯 × 256 B, 100 ms flush; FsWatcher coalesce 100 ms; Selection summary debounce 80 ms; LowMemory re-arm 5 s; EmptyWorkingSet throttle 1 s.
  • FileEntry sizeof: 정확히 40 B (static_assert, design.md:153, 399).
  • 측정 미달 (Amber): ① LVN_GETDISPINFO p99 정량값 — 매뉴얼 1-hour soak 미실행 ② EmptyWorkingSet 200 ms 복구 게이트 — UI 자동화 필요 ③ dual-pane 50-nav 10 MB drift — 30분 매뉴얼 미실행. WinUI 3 비교 시 이 3개는 baseline number를 비교하는 게 아니라 "동일 protocol에서 신규 측정" 으로 처리해야 함.

  • ListView는 데이터 가상화(random-access)와 UI 가상화를 함께 지원하며, "100만 항목 컬렉션에서 100,000–100,050만 로드, 사용자가 처음으로 이동하면 1–50만 로드" 식의 스파스 액세스 패턴이 공식 가이드 문서에 명시되어 있다 [Learn data virtualization].
  • ItemsRepeater는 가상화는 하지만, "ItemsSource에 큰 컬렉션을 할당하는 것만으로 10초 이상 소요"되는 알려진 결함이 있다(InspectingDataSource::WrapIterable이 AnnotatedScrollBarLabel 컬렉션을 전수 순회) [#8888 Closed]. 100k 데이터셋을 다루는 fast-explorer 시나리오에서 직접 사용은 위험.
  • ItemsView는 사실상 ItemsRepeater의 상위 래퍼이므로 동일 결함의 영향권일 수 있으나, ScrollView/InteractionTracker 통합으로 스크롤 부드러움이 향상된다 [Learn ItemsView].
  • 비균등 항목 높이(variable row height)는 ItemsRepeater 가상화 가정과 충돌하며 layout pass 비용이 급증하여 "10 fps 미만"으로 떨어진 사례가 보고됨 [Discussion #7514]. fast-explorer 듀얼-페인은 균일 행 높이라 이 결함은 회피 가능.

B.2 컨테이너 재활용과 ContainerContentChanging

WinUI 3 ListView/GridView의 핵심 가상화 메커니즘은 container recycling이다. 스크롤로 화면 밖으로 나간 ListViewItem은 폐기되지 않고 recycle queue에 들어가 새 데이터에 재바인딩된다. Win32 LVS_OWNERDATA의 LVN_GETDISPINFO 콜백과 개념적으로 유사하나, 컨테이너 재사용 비용이 0이 아니라는 점에서 차이가 있다.

  • ContainerContentChanging 이벤트는 컨테이너가 새 데이터로 채워지거나 recycle queue에 들어갈 때마다 발생하며, InRecycleQueue/Phase/ItemIndex 인자를 제공한다. Phase 0에서 placeholder/Opacity 0 처리 후, RegisterUpdateCallback으로 다음 phase를 예약하는 점진적 렌더링 패턴이 권장된다 [Learn ContainerContentChanging].
  • 최대 성능을 위해서는 마크업 바인딩을 제거하고 코드비하인드에서 직접 값을 대입해야 한다(공식 권고) [Learn Optimize ListView]. — 이는 x:Bind보다도 더 절약 가능하다는 의미.
  • 전제 조건: ContainerContentChanging은 ItemsPanel이 ItemsStackPanel 또는 ItemsWrapGrid일 때만 발생한다. 패널을 교체하면 이벤트가 사라지므로 fast-explorer가 커스텀 패널을 쓰려면 별도 진행률 렌더링 전략이 필요 [Learn Optimize ListView].
  • 패널 캐시 길이: ItemsStackPanel.CacheLength / ItemsWrapGrid.CacheLength 기본값은 데스크톱 4, 모바일 2. 4 = "뷰포트의 4배"만큼 선행/후행 컨테이너를 materialize → 100k 행 중 약 viewport*5 개만 실제 트리에 존재 [ListView basics & virtualization].
  • GetContainerForItemOverride 비용: 새 컨테이너 생성 시 호출되며, recycling이 깨지면 매 행마다 호출되어 가상화가 사실상 무력화된다 — ChoosingItemContainer·ItemTemplateSelector 오용으로 발생하는 회귀가 다수 보고됨 [#9307 closed] [#9189 closed].

fast-explorer 매핑: 현재의 113-슬롯 dummy vtable + IListView_FE::SetOwnerDataCallback 트릭은 WinUI 3에는 직접 대응이 없다. 대신 (a) ContainerContentChanging Phase 분할 + (b) Phase 0에서 텍스트/이름만 채우고 (c) Phase 1+에서 아이콘·확장자 색상 등을 점진 채우는 패턴으로 같은 효과(불필요한 ImageList 조회 억제)를 달성할 수 있다.

B.3 DataTemplate 선택과 x:Bind / x:Phase

x:Bind vs Binding

  • x:Bind는 컴파일 타임에 partial class 코드를 생성해 reflection 없이 직접 프로퍼티 접근. 공식 문서는 "compile-time validation, breakpoint 가능, 우수한 성능" 으로 권장 [Learn x:Bind].
  • {Binding}은 런타임 reflection 기반. 커뮤니티 자료(주의: 정량 수치 출처는 비공식 블로그)에 따르면 property access가 최대 5배 정도 빠르다고 보고되나, Microsoft 공식 벤치마크는 발견되지 않음 [chriscodes blog, non-Microsoft].
  • x:Bind 기본 모드는 OneTime(Binding은 OneWay). OneTime은 change-detection 훅 코드를 생성하지 않아 더 가볍다 [Learn x:Bind markup].

x:Phase (점진 렌더링)

  • x:Phase는 x:Bind 바인딩과 함께 쓰여, "스크롤 중에는 placeholder만 그리고, 사용자가 멈추면 후속 phase를 렌더링" 하는 점진 업데이트를 마크업만으로 표현 가능. WinUI 3에서도 동일하게 동작한다 [Learn Optimize ListView].
  • 한계: Phase는 x:Bind 전용. {Binding} 또는 코드비하인드 setter에는 적용되지 않는다. 또한 Phase는 그려질 순서를 약속할 뿐, 실제 binding 실행은 idle 시점에 합쳐질 수 있어 정밀한 시간 제어는 불가 (공식 문서가 구체 latency 보장을 제시하지 않음) — no public number found.

ItemTemplateSelector vs ChoosingItemContainer

  • ItemTemplateSelector는 최대 5개의 후보 컨테이너만 비교해서 재사용 여부를 결정한다 — fast-explorer가 폴더/파일/심볼릭링크/숨김파일 등 다중 템플릿을 쓰면 5개를 넘는 순간 가상화가 깨질 위험 [Learn Optimize ListView].
  • WinUI 3에서 ItemTemplateSelector는 ListView/GridView/TreeView에서 컨테이너 재활용 시 호환 안 되는 컨테이너를 선택해 깨지는 결함이 보고됨 [#9189 closed]. ChoosingItemContainer이벤트 사용이 권장 경로.

B.4 이미지/아이콘 렌더링 비용

fast-explorer는 행마다 16×16 또는 32×32 셸 아이콘을 표시한다. WinUI 3 셀에서 이를 구현할 때의 비용:

  • BitmapImage: 디코딩과 GPU 업로드를 XAML이 자동 수행. DecodePixelWidth / DecodePixelHeight를 설정하지 않으면 풀 사이즈로 캐싱되어 메모리 폭발(예: 3800×3000 JPG → ~43 MB 디코드, 10 MB 원본 대비 4배) [Learn Images]. 행마다 다른 아이콘이라도 16/32px decode를 강제하는 것이 필수.
  • StorageFile.GetThumbnailAsync: Windows 썸네일 디스크 캐시(thumbcache_*.db)를 이용해 가장 빠른 경로. OneDrive 클라우드-온리 파일은 placeholder만 반환됨 [Learn GetThumbnailAsync] [#1874 OneDrive thumbnail]. fast-explorer가 기존에 사용하는 IShellItemImageFactory::GetImage 경로(C++ COM)는 보존하는 편이 비용·동작 모두 우월.
  • WinUI에는 WPF의 Imaging.CreateBitmapSourceFromHIcon에 1:1 대응되는 API가 없음. HICON/HBITMAP을 XAML 이미지 소스로 올리려면 픽셀을 메모리 스트림으로 복사 후 SoftwareBitmap으로 래핑하는 추가 비용 발생 [Discussion #7943].
  • SoftwareBitmapSource: BGRA8 픽셀 버퍼를 직접 받아 GPU 업로드. BitmapImage보다 디코딩 단계가 없어 초기 비용은 낮지만, 매 행마다 IPC를 거쳐 셸로부터 비트맵을 받는 디자인이면 결국 동일 — caching 전략이 핵심.
  • 권고: C++ 코어가 만든 HBITMAP ImageList 캐시를 유지하고, WinUI 행은 thumbnail key(예: file-type SHFILEINFO icon index)를 통해 메모리 매핑된 동일 SoftwareBitmap을 공유. row당 신규 alloc 금지.

B.5 스크롤·합성(Composition) 파이프라인

  • XAML ScrollViewer 는 UI 스레드에 종속. 큰 layout pass(예: variable-height items, heavy template)가 UI 스레드를 점유하면 스크롤이 즉시 stutter. Microsoft 공식 문서: "frame이 길게 실행되면 디스플레이 업데이트뿐 아니라 input 처리도 막힌다" [Learn WinUI perf].
  • InteractionTracker는 독립 input 스레드에서 manipulation을 처리해 UI 스레드 부하와 무관하게 60+ fps 스크롤을 유지. ScrollView(신규 컨트롤)와 ItemsView는 내부적으로 InteractionTracker를 사용 [Learn Input-driven animations] [#108 ScrollView spec, Open].
  • Composition thread는 UI 스레드와 분리되어 visual tree의 transform·opacity·brush 등을 합성. 그러나 컨텐츠 변경(텍스트 변경, 신규 컨테이너 materialize)은 여전히 UI 스레드 통과 — 가상화 콜백 비용이 16.6 ms 예산을 넘으면 빈 칸이 보인다 [Learn Visual layer].
  • 알려진 stutter 원인:
    • WinUI 3 빈 윈도우에서 프레임당 29–31 ms 측정(UWP는 ~0.5 ms) — 60×차이 보고 [#6031 closed]. (현재는 회귀 수정됨)
    • 마우스 이벤트는 ~1000 Hz로 발사되지만 UI 함수 실행은 ~30 fps에 cap되는 회귀 [#7290 closed, fixed in WinAppSDK 1.3].
    • 고주사율 모니터에서 애니메이션이 lag/non-smooth [#7373 closed].
    • WinUI 3 Win32에서 정밀 터치패드 scrolling personality가 UWP 대비 열등(bounce·glide 누락) [#2513 closed].
  • 120 Hz 대응: WinUI 3는 명시적 120 Hz 보장 문구가 없으며, 위 회귀 사례들로 보아 vsync-aligned high-refresh-rate 스크롤은 보장되지 않는다. 현재 Win32 ListView도 GDI/DWM에 의존하므로 동일하나, fast-explorer가 "60–120 Hz 스크롤"을 요구사항으로 두면 InteractionTracker 기반 컨트롤(ItemsView/ScrollView) 채택이 필수.

B.6 시작 시간 / 메모리 풋프린트

  • Native AOT (Windows App SDK 1.6+): Microsoft 샘플 Contoso Camera 앱 측정 결과 start time 50% 단축, 패키지 크기 framework-package 시 ~8×, self-contained 시 ~2× 감소 [Windows Dev Blog WASDK 1.6]. 단, "모든 타입이 trimming-safe하게 root되도록 개발자가 보장해야 함" — reflection 기반 {Binding} 타깃이 위험 (1.6 단계에서는 자동화 없음).
  • WinUI 3 unpackaged 앱 메모리: 최소 템플릿 앱에서도 추가 메모리 비용이 크다는 이슈가 Open 상태로 추적 중 [#8736 Open]. 정확한 수치는 스크린샷에만 존재(텍스트로 캡처 안 됨) — no clean public number found.
  • Win32 비교 직접 수치: Microsoft Learn의 startup performance 문서는 절대 ms 수치를 공개하지 않는다 [Learn startup perf]. 일부 3rd-party 블로그/뉴스가 File Explorer 콜드 launch 2.7 s → 1.4 s, Notepad 1.1 s → 0.4 s 등을 인용하나, 출처가 windowsnews.ai 같은 2차 매체이므로 본 보고에서는 "1차 소스 미확인, 참고 수치"로 기록한다 [windowsnews.ai 2차].
  • 실측 비교점(fast-explorer 관점): 현재 Win32 fast-explorer는 모듈 자체가 native이며 cold launch는 통상 100 ms 이내(가정). WinUI 3 + .NET 8 unpackaged + AOT 미적용 시 초기 XAML 파싱/JIT 비용으로 500 ms~1.5 s 추가 부담이 예상되며, AOT 적용 시 절반 정도 회수 가능하다는 시나리오가 합리적 (직접 측정치 없음).

  • (2) Win32 LVS_OWNERDATA의 직접 대응: ContainerContentChanging + ChoosingItemContainer + ItemsStackPanel 조합. LVN_GETDISPINFO의 "필요한 행만 조회"는 가상화 + Phase 분할로 재현 가능하지만, "113-슬롯 dummy vtable로 ImageList 콜백 억제"같은 마이크로-최적화는 WinUI 3에 등가물이 없고 ContainerContentChanging Phase로 같은 효과를 더 안전하게 달성 [Learn Optimize ListView].
  • (3) 바인딩: 모든 셀에서 x:Bind OneTime 사용 필수. {Binding}은 reflection으로 100k 행에서 비용이 비선형 증가. 더 빠른 경로는 ContainerContentChanging에서 코드비하인드로 setter를 직접 호출 (Microsoft 공식 "maximum performance" 권고) [Learn Optimize ListView].
  • (4) DataTemplate selector 위험: ItemTemplateSelector는 후보 5개 한계 + WinUI 3 회귀(#9189, #9307)로 fast-explorer의 다중 항목 타입(폴더/파일/링크)에 쓰면 가상화가 깨질 가능성이 높음. ChoosingItemContainer이벤트로 대체 권장 [#9189].
  • (5) 아이콘: WinUI에는 CreateBitmapSourceFromHIcon 1:1 대응 API 없음. StorageFile.GetThumbnailAsync은 OneDrive 클라우드-온리에서 placeholder만 반환하고, 시스템 썸네일 캐시 사이즈에 제한. C++ 코어의 기존 IShellItemImageFactory+HBITMAP 캐시를 유지하고 SoftwareBitmapSource로 WinUI에 어댑팅하는 것이 비용·동작 모두 최적 [Discussion #7943] [#1874].
  • (6) 스크롤 60–120 Hz: 전통적 XAML ScrollViewer는 UI 스레드 종속이라 layout 부하 시 stutter. ItemsView/ScrollView가 사용하는 InteractionTracker가 독립 스레드에서 manipulation 처리 → fast-explorer의 120 Hz 요구를 충족하려면 이 경로가 사실상 필수. 그러나 WinUI 3 자체에 ~30 fps cap 회귀(#7290), Win32 정밀 터치패드 열등(#2513) 이력이 있어 회귀 검증 부담이 큼 [Learn] [#7290].
  • (7) 시작 시간: Native AOT(WASDK 1.6+)로 Microsoft 자체 측정 -50% (Contoso Camera). 그러나 trimming-safe 강제 + {Binding} reflection 위험 → fast-explorer는 x:Bind 일관 적용 시 AOT 채택 가능 [Win Dev Blog WASDK 1.6].
  • (8) 메모리: WinUI 3 빈 윈도우조차 unpackaged 모드에서 비대한 메모리 footprint가 Open 이슈로 추적 중(#8736). fast-explorer가 현재 native Win32로 수 MB 수준이라면 WinUI 3 전환 시 최소 수십 MB 증가 예상(공식 1차 수치 없음).
  • (9) Files 앱 reality check: 동일 도메인 mature WinUI 3 앱이 폴더 진입 1–3 s, hover 100–200 ms, 2000+ 항목 unresponsive 회귀를 반복적으로 겪었다는 사실은 100k 항목·120 Hz 목표를 WinUI 3로 달성하기가 "가능은 하나 평범한 구현으로는 회귀 위험"임을 시사한다 [Files#14939] [Files#15554].
  • (10) 권장 검증 전략(C/D 에이전트에게): POC에서 (a) 100k 더미 row에 ListView+x:Bind OneTime+ContainerContentChanging Phase, (b) 동일 데이터에 ItemsView, (c) 동일 데이터에 ItemsRepeater+ScrollView, 세 변종을 모두 측정. 측정 지표: 콜드/웜 launch ms, ItemsSource set 후 첫 paint까지 ms, 60·120 Hz 모니터에서 wheel·flick·drag 스크롤 fps, 1000 row 빠른 스크롤 시 dropped frames %.

C.1 윈도우/HWND 모델 호환성

현재 MainWindow는 전형적인 Win32 child-HWND 트리 구조이다. src/ui/main-window.cpp:1383 의 CreateWindowExW(0, kClassName, L"Fast Explorer", WS_OVERLAPPEDWINDOW, ...) 가 최상위 프레임 HWND를 만들고, 그 자식으로:

  • 상태바 1개 (STATUSCLASSNAMEW, main-window.cpp:87–90)
  • 슬롯당 1개씩 최대 4개의 toolbar-row (std::array<std::unique_ptr<PaneToolbarRow>, 4>, main-window.h:259)
  • 슬롯당 1개의 Edit 주소바 + BS_OWNERDRAW 드롭다운 버튼 (main-window.cpp:105–109, 130–136)
  • 슬롯당 1개의 WC_LISTVIEWW 리포트뷰 (main-window.cpp:146–150, listViews_[4])
  • 스플리터 3개 (splitterHwnds_[3], main-window.h:294)

배치는 MainWindow::relayout() (main-window.cpp:490–587)이 WM_SIZE에서 SetWindowPos로 직접 좌표를 갱신한다. 즉 UI는 14–15개의 별도 HWND가 살아 있는 트리이며, 각 HWND는 자기 wndProc로 메시지를 받는다.

WinUI 3에서 최상위 윈도우는 Microsoft.UI.Xaml.Window이며, 그 backing HWND는 IWindowNative::get_WindowHandle (GUID EECDBF0E-BAE9-4CB6-A68E-9598E1CB57BB)으로 얻을 수 있다 (Microsoft Learn — Retrieve a window handle (HWND), microsoft-ui-xaml#5151). AppWindow는 그 HWND를 WindowId로 변환해 (Microsoft::UI::GetWindowIdFromWindow) 얻고 AppWindow::GetFromWindowId 로 가져온다 (Microsoft Learn — GetWindowIdFromWindow).

문제는 해당 HWND 안의 콘텐츠다. WinUI 3 데스크톱 윈도우의 HWND는 그 자체가 ContentIsland(DirectComposition 기반 합성 표면)에 직접 부착돼 있고, 그 위에 Win32 자식 HWND(WC_LISTVIEW, STATUSCLASSNAME, 스플리터 등)를 SetParent 또는 CreateWindowExW(... parent=xamlHwnd ...) 로 얹는 길은 공식적으로 막혀 있다. microsoft-ui-xaml#10050(ThousandIslands Window 제안)은 이 한계를 다음과 같이 명시한다: “The XAML island is the top-level window itself ... putting custom Win32 windows as children of the window never worked out well: this way, you are essentially putting content INSIDE an existing island ... it's possible to show something in a child window but it's not possible to get input working properly (keyboard, mouse, pointer).” (microsoft-ui-xaml#10050).

따라서 현재의 4-pane HWND 트리(listview × 4, toolbar row × 4, address Edit × 4, splitter × 3, status bar)는 그대로 옮길 수 없다. 두 가지 선택지가 있다:

  1. 전면 재설계: ListView → XAML ListView/ItemsRepeater로 치환, 스플리터 → XAML Grid + GridSplitter, 주소바 → XAML AutoSuggestBox 등. 이 경로에서 main-window.cpp(3,242 LOC)와 pane-toolbar-row, pane-splitter, address-bar-popup, listview-group-callback, dispinfo-histogram, label-edit-controller, dark-scrollbar-hook, selection-sync는 거의 전량 재작성 대상.
  2. XAML Islands 역방향: Win32 프레임 안에 XAML 콘텐츠를 부분 호스팅하는 길. 그러나 데스크톱 WinUI 3에서 WindowsXamlManager 기반 XAML Islands는 “deprecated path” 로 분류되며 본 시나리오(WinUI 3 전환)와는 방향이 반대다. (WindowsAppSDK#465 — XAML Islands 위치).

결론: HWND 4개짜리 pane 아키텍처는 사실상 살아남지 못한다. 유일하게 보존되는 HWND는 최상위 프레임 HWND 1개이며, 거기에는 OLE(RegisterDragDrop) 가입과 shell 컨텍스트 메뉴 owner-window 지정 정도만 걸 수 있다.

C.3 Drag & Drop 인터롭 비용

현재 코드는 OLE drag-drop을 두 방향 모두 직접 다룬다:

  • Drop in: 각 listview HWND에 PaneDropTarget을 RegisterDragDrop 등록 (main-window.cpp:617–624, main-window.cpp:1676). PaneDropTarget::DragEnter/Over/Leave/Drop 안에서 ListView_HitTest로 행을 찾고 (drop-target.cpp:66–94), 해당 폴더의 shell IDropTarget을 caching해서 drop effect 계산을 통째로 shell에 위임 (drop-target.cpp:102–142).
  • Paste (clipboard → shell drop): OleGetClipboard → 폴더의 shell IDropTarget → DragEnter/DragOver/Drop 시퀀스를 직접 시뮬레이션 (clipboard-ops.cpp:89–117).

WinUI 3 에서는 두 갈래의 경로가 공존한다:

  1. XAML 네이티브 경로: UIElement.AllowDrop = true + DragOver/Drop 이벤트. payload는 DataPackageView(read) / DataPackage(write)로 표현되며, shell CFSTR_FILECONTENTS 같은 형식을 표현하기 위한 공식 변환층은 없다 (Microsoft Q&A — DataPackage with shell resources). microsoft-ui-xaml#10119은 “Explorer에서 파일을 끌어왔을 때 DragOver가 안 뜨고 not-allowed 커서가 뜨는” 회귀를 확인된 버그(#8108 중복)로 문서화한다.
  2. Win32 OLE 경로 유지: RegisterDragDrop(frameHwnd, dropTarget)을 그대로 호출. 이 경로에서 IDropTarget::DragEnter가 받는 것은 우리 코드가 익숙한 raw IDataObject*이다. 문제는: (a) 등록은 프레임 HWND 1개에만 가능 — XAML이 자기 wndProc 안에서 OLE 메시지를 소비하는지 문서화된 동작 불명, (b) hit-test가 XAML 트리 좌표계에서 일어나야 하므로 drop-target.cpp:64–87의 ListView_HitTest는 통째로 XAML VisualTreeHelper.FindElementsInHostCoordinates 같은 API로 다시 써야 한다.
  3. CoreDragDropManager / IDragDropManagerInterop 경로: IDragDropManagerInterop::GetForWindow(hwnd, IID_ICoreDragDropManager)로 HWND에 매여 있는 CoreDragDropManager를 받아서 TargetRequested 이벤트에 hook하는 방식. Windows 11+ 한정이고 (sdk-api — IDragDropManagerInterop::GetForWindow), Drag source 측 visuals만 컨트롤하는 데 적합하다 — shell IDropTarget 전체 위임 모델로는 부족.

현실적 권장: 프레임 HWND 1개에 RegisterDragDrop으로 OLE drop target 1개 유지하되, pane별 hit-test 로직(drop-target.cpp:60–94)을 XAML 좌표계 hit-test로 재작성. 이 경로는 “정상 동작 문서화 불명”이며 (#10119의 회귀가 OLE 경로에도 영향을 미치는지 별도 검증 필요). Drag source(FileDropSource + DoDragDrop)는 XAML PointerPressed에서 직접 호출 가능 — shell IDataObject를 만들고 (clipboard-ops.cpp:14–39 재사용), DoDragDrop 호출. XAML DragStarting 경로로 옮기면 DataPackage로 형식을 옮겨야 하므로 shell-호환 IDataObject를 잃고, Explorer 같은 외부 앱이 우리 드래그를 받아주지 않을 위험이 있다 (문서화 불명).

C.4 컨텍스트 메뉴 호스팅

현재 컨텍스트 메뉴는 전형적인 Win32 owner-draw 패턴이다 (shell-context-menu.cpp:204–292):

  1. CreatePopupMenu() + IContextMenu::QueryContextMenu 로 HMENU 채움.
  2. caller가 넘긴 ExtraSubmenu를 AppendMenuW(MF_POPUP, ...)로 hierarchy에 매단다 (:223–239).
  3. SubclassGuard (:58–74) 로 owner HWND를 임시 subclass — WM_INITMENUPOPUP / WM_DRAWITEM / WM_MEASUREITEM / WM_MENUCHAR 4종 메시지를 IContextMenu3::HandleMenuMsg2 (없으면 IContextMenu2::HandleMenuMsg) 로 forward.
  4. TrackPopupMenuEx(... TPM_RETURNCMD ...) 로 modal popup — 사용자 선택을 반환받음.
  5. cmd가 kCmdIdMax(0x7FFF)보다 크면 caller의 extra submenu → PostMessageW(owner, WM_COMMAND, ...). 작으면 shell verb → IContextMenu::InvokeCommand 혹은 per-item fan-out (:184–202).

이 코드를 WinUI 3 안에서 그대로 실행할 수 있는가? 두 측면을 분리해야 한다:

  • TrackPopupMenuEx 자체는 Win32 API이고 HWND만 있으면 동작한다. 프레임 HWND를 IWindowNative::get_WindowHandle로 얻어 그대로 owner로 넘기면 메뉴 자체는 뜬다. 이는 Win32 API 호출이므로 XAML 합성 표면과 무관 — OS가 별도 popup HWND를 만든다.
  • 그러나 WM_INITMENUPOPUP / WM_DRAWITEM / WM_MEASUREITEM 이 우리 wndProc로 도달하는지가 핵심. 이 메시지들은 owner HWND의 wndProc로 전달되는데, WinUI 3 프레임 HWND의 wndProc는 Microsoft.UI.Xaml 내부 코드가 소유한다. SetWindowSubclass 자체는 호출 가능하지만, XAML 내부 wndProc가 이 메시지들을 사용자 subclass에 전달하기 전에 소비할 가능성에 대한 공식 문서는 없다 (검색 결과 microsoft-ui-xaml 저장소 어디에도 owner-draw shell menu 패턴 가이드 없음; #6925 — flyout이 윈도우 밖으로 나갈 수 있는가도 해결되지 않은 채로 열려 있음).

대안은 전면 MenuFlyout 재구성이다 — IContextMenu::QueryContextMenu 로 HMENU 를 채운 뒤, HMENU를 walk하면서 각 항목(GetMenuItemInfoW)을 MenuFlyoutItem으로 옮기고, 사용자가 클릭하면 IContextMenu::InvokeCommand 를 직접 호출. 이 경로의 약점:

  • shell extension의 owner-draw 항목(예: “Open With” 아이콘 + 부제목, “Share” 미니 미리보기)이 모두 그림 없이 텍스트만 남는다. MF_OWNERDRAW를 받아서 그릴 device context가 MenuFlyout에는 없다.
  • WM_MENUCHAR hotkey 라우팅 (shell-context-menu.cpp:30–37)이 사라지므로 Office 같은 서드파티 shell ext의 단축키가 안 먹는다.
  • shell 컨텍스트 메뉴의 깊은 nested submenu(예: “Send to → ...”)는 동적으로 채워지는데, 이건 HMENU 위에서 WM_INITMENUPOPUP으로 lazy-populate 되는 메커니즘이라 MenuFlyout으로 옮기면 동적 채움 시점이 어긋난다.

결론: Win32 HMENU + TrackPopupMenuEx 경로를 유지하는 것이 기능적으로 안전하다. 프레임 HWND를 owner로 쓰는 것은 가능, 그러나 subclass-forward 패턴이 XAML wndProc 안에서 살아남는지는 검증 필요 항목으로 남는다.

C.5 아이콘 파이프라인 (HICON → XAML ImageSource)

현재 아이콘 파이프라인은 다음 4단계다:

  1. IconCache::IconCache(dpi) → createPlaceholderImageList(dpi) 로 HIMAGELIST를 만들고 2개 placeholder icon을 채움 (icon-cache.cpp:34–61).
  2. list-view에 ListView_SetImageList + LVS_SHAREIMAGELISTS 부착.
  3. extension별로 STA worker에서 SHGetFileInfoW(SHGFI_ICON|SHGFI_SMALLICON|SHGFI_USEFILEATTRIBUTES) 호출 (icon-provider.cpp:63–81) → HICON 반환.
  4. UI 스레드에서 ImageList_AddIcon으로 slot index 받음 → ExtensionIconCache(extension-icon-cache.h:28–64)가 extension → slot index 매핑.

XAML ListView / ItemsRepeater 에는 HIMAGELIST 개념이 없다 — 각 아이템 row가 Image 컨트롤을 들고, Image.Source는 ImageSource(BitmapImage / WriteableBitmap / SoftwareBitmapSource) 중 하나여야 한다. HICON → ImageSource 직결 API는 없다.

공개된 변환 경로:

  1. HICON → HBITMAP → SoftwareBitmap → SoftwareBitmapSource 경로 (microsoft-ui-xaml#7943): GetIconInfo로 HBITMAP 추출 → GetDIBits로 픽셀 추출 → SoftwareBitmap 생성 → SoftwareBitmapSource.SetBitmapAsync. 16×16 BGRA8 기준 한 아이콘당 1024 byte memcpy + GPU upload 1회.
  2. StorageFile.GetThumbnailAsync (#7943에서 권장된 길): 우리 PIDL 경로 모델과 충돌한다 — StorageFile은 권한이 있는 실제 파일 경로를 요구하고 비동기다. 현재 우리 cache 키는 extension 단위(extension-icon-cache.h:13–15)이므로 fit하지 않다.
  3. SoftwareBitmapSource의 호환성: WinUI 3의 Microsoft.UI.Xaml.Media.Imaging.SoftwareBitmapSource는 WinRT API이고 UI 스레드에서 SetBitmapAsync 호출이 필요 (Microsoft Learn — SoftwareBitmapSource).

비용 추산 (개별 검증 필요, 추정치): small icon 16×16 BGRA8 한 장 = 1 KB. ExtensionIconCache::kDefaultCapacity = 256 (extension-icon-cache.h:30) 가득 채우면 256 KB의 CPU 사이드 버퍼 + 256개의 SoftwareBitmap + 256번의 SetBitmapAsync 호출. GPU upload는 XAML 렌더러가 lazy로 한다고 알려져 있으나 정확한 시점은 문서화되지 않음.

cache key 영향: 현재 key는 extension(소문자 정규화, extension-icon-cache.h:36–47), value는 HIMAGELIST slot index(int). XAML 경로에서는 value를 winrt::Microsoft::UI::Xaml::Media::ImageSource 같은 RAII 핸들로 바꿔야 한다 — int → smart pointer로 바뀌므로 extension-icon-cache.{h,cpp}는 시그니처 수정 필요 (key 의미는 유지). LRU 정책(std::list) 자체는 재사용 가능.

icon-cache.cpp:80–99의 byteSize() diagnostic은 의미가 바뀐다 — HIMAGELIST가 사라지므로 SoftwareBitmap 풀의 추정 크기를 새로 계산해야 한다.

C.6 스레딩 모델 매핑

현재 스레딩 토폴로지:

  • 메인 UI 스레드: OleInitialize (main.cpp:64) → STA + OLE. message loop는 main.cpp:168–192 의 GetMessageW + DispatchMessageW 루프.
  • ShellWorker: std::jthread + CoInitializeEx(COINIT_APARTMENTTHREADED) (shell-worker.cpp:230–251). IFileOperation 요구사항이라 STA 필수.
  • IconProvider: 동일 패턴 — std::jthread + STA (icon-provider.cpp:95–113).
  • FS watcher / WinSparkle 백그라운드: 자세히 검증 필요 — 본 보고 범위 밖.

WinUI 3에서는 Microsoft.UI.Dispatching.DispatcherQueue가 UI 스레드의 디스패처 역할을 한다. 문서상 WinUI 3 UI 스레드는 “STA UI model — all UI elements are owned by the thread that created them and are backed by a DispatcherQueue tied to that thread.” (Microsoft Learn — Threading functionality migration). DispatcherQueueController는 DQTYPE_THREAD_CURRENT + DQTAT_COM_STA 로 만들 수 있다고 명시 (DispatcherQueueUpdates spec).

호환성:

  • UI 스레드 STA → 우리 OLE/Shell COM 요구와 정확히 일치.
  • Worker STA jthread (shell-worker, icon-provider) → UI 스레드와 무관하게 자기 apartment 운영, 변경 불요.
  • 현재 UI ← worker 통신은 PostMessageW(host, kWmFe*, ...) 기반의 ResultChannel (icon-provider.h:81, shell-worker.h:60–100). WinUI 3에서는 프레임 HWND가 살아 있지만 그 wndProc가 우리 것이 아니므로 사용자 메시지가 어디로 가는지 검증 필요 — 권장 대안은 DispatcherQueue.TryEnqueue 로의 마이그레이션 (Microsoft Learn — DispatcherQueue).
  • 주의: WinUI 3 STA 는 UWP의 ASTA (non-reentrant) 보장이 없다 — “the Windows App SDK's STA threading model doesn't have the same assurances around preventing reentrancy issues”. 현재 코드가 ASTA 가정에 의존하지는 않지만(우리는 본래 Win32 STA), 새 코드 작성 시 주의.

C.7 빌드/바이너리 크기 영향

  • WindowsAppRuntime 의존성: WinUI 3는 standalone 배포가 아니라 WindowsAppRuntime framework package에 의존한다. 자체 포함 배포(WindowsAppSDKSelfContained=true) 선택 시 앱과 함께 ship해야 하는 리디스트리뷰터블은 x64 .msix 약 20.3 MiB, executable redistributable 약 56.8 MiB 수준 (microsoft-ui-xaml#7683 — Hello World size). 현재 fast-explorer는 WinSparkle 외 추가 런타임 의존성이 없는 단일 실행파일이므로 배포 사이즈가 한 자릿수 MiB → 두 자릿수 MiB로 점프한다.
  • 컴파일 타임: C++/WinRT 헤더(winrt/Microsoft.UI.Xaml.h, winrt/Microsoft.UI.Xaml.Controls.h 등)는 파싱 자체가 무겁다 — Microsoft 자체 측정에서 winrtHeaders.h 한 파일이 16.4초 빌드의 8.6초(52.3%)를 차지한 사례 (Microsoft Learn — Build Insights). PCH로 winrt headers 를 묶는 것이 필수.
  • C++/WinRT 자체 코드 크기: 헤더-온리 projection이므로 사용 안한 type은 코드 생성되지 않음. Microsoft 측 측정 — C++/CX 대비 binary 20% 감소 (Microsoft Learn — What's new in C++/WinRT). 즉 C++/WinRT 자체는 부담이 작고, 부담은 WindowsAppRuntime DLL이 차지한다.
  • WinSparkle(app/main.cpp:108–134)와 충돌하는가? 별도 검증 필요. WinSparkle은 자기 STA + 백그라운드 스레드를 들고 있고 WinUI 3 DispatcherQueue와 무관하므로 이론적으로는 공존 가능. 문서화된 호환성 확인은 본 보고 범위 밖.

C.9 C++/WinRT 시나리오 핵심 요약

  • Shell COM 호출 코드는 거의 전량 재사용 가능. shell-bind, shell-worker, drop-source, clipboard-ops, com-raii 5개 파일(704 LOC)이 그대로 살아남는다. C++/WinRT는 ABI 수준에서 그저 COM이므로 raw IShellFolder*, IContextMenu*, IDropTarget*, IFileOperation*는 전부 그대로 호출 가능.
  • HWND 트리는 살아남지 못한다. WinUI 3 윈도우의 backing HWND는 IWindowNative::get_WindowHandle로 얻을 수 있지만 (Retrieve HWND), 그 HWND 아래에 우리의 4개 listview/3개 splitter/4개 toolbar-row HWND를 자식으로 매다는 길은 Microsoft에서 명시적으로 “input이 동작하지 않는다”고 인정한 상태 (microsoft-ui-xaml#10050). → main-window.cpp의 3,242 LOC는 ~85% 재작성.
  • Drop target은 frame HWND 단일 등록으로 축소. RegisterDragDrop으로 OLE 경로를 살려두면 shell IDataObject 위임 모델이 그대로 동작할 가능성이 있으나, Explorer → WinUI 3 drag-in 자체가 회귀 버그(#10119/#8108)로 문서화돼 있어 검증 필요. pane 별 hit-test (drop-target.cpp:60–94)는 XAML 좌표계로 재작성.
  • 컨텍스트 메뉴는 두 갈래의 위험: TrackPopupMenuEx + owner subclass 경로가 살아남는지 문서화된 보장 없음; 살지 않으면 MenuFlyout으로 옮겨야 하는데 shell extension의 owner-draw 항목(아이콘, “Open With” 부제 등)과 WM_INITMENUPOPUP 기반 lazy submenu 채움이 깨진다. → shell-context-menu.cpp 376 LOC 중 핵심 호출부(~200)는 유지, owner subclass + Track 부분은 검증 후 결정.
  • 아이콘 파이프라인은 새 비용. HIMAGELIST는 XAML에서 사용 불가 — HICON→HBITMAP→SoftwareBitmap→SoftwareBitmapSource 변환층 신설 필요 (microsoft-ui-xaml#7943). icon-cache.cpp(107 LOC)는 거의 전량 재작성, extension-icon-cache의 LRU 구조는 유지.
  • 스레딩 모델은 호환. UI 스레드 STA + DispatcherQueue (DQTAT_COM_STA) 가 현재의 OleInitialize STA 와 정확히 일치 (Threading migration). ShellWorker / IconProvider 의 STA jthread 패턴(shell-worker.cpp:230–251, icon-provider.cpp:95–113)은 변경 불요. UI ← worker 통신만 PostMessageW → DispatcherQueue.TryEnqueue로 마이그레이션 권장 (host wndProc 이 우리 것이 아니므로).
  • 배포 사이즈가 점프. WindowsAppRuntime framework 의존 → self-contained x64 .msix ~20 MiB / executable redistributable ~57 MiB (#7683). 현재 단일 .exe + WinSparkle 모델 대비 ship size가 크게 늘어남.
  • 컴파일 타임 비용. C++/WinRT 헤더는 Microsoft 측정 기준 빌드의 ~52%를 차지 (Build Insights); PCH 필수.
  • 총평 (추정): 본 보고가 직접 검토한 11개 파일 5,485 LOC 기준 재사용 ~37% / 재작성 ~63%. 재작성 부담은 거의 전부 main-window.cpp 및 pane UI 컨트롤군에 집중. core/* 와 shell COM 래퍼군은 명세대로 살아남는다. 가장 큰 위험 두 가지는 (1) shell 컨텍스트 메뉴의 owner-draw 메시지가 XAML wndProc 안에서 forward 되는지 문서화 불명, (2) Explorer → WinUI 3 drag-in 회귀 (#10119/#8108).

D.2 마샬링 비용 모델 (LibraryImport / GeneratedComInterface)

.NET 7 이상에서는 두 개의 source generator가 인터롭 경로를 크게 바꿨다.

  • [LibraryImport] (P/Invoke source generator): classic [DllImport]은 런타임에 IL stub을 생성하고 JIT한다. [LibraryImport]는 컴파일 시점에 marshalling C# 코드를 생성하므로 runtime IL stub 생성이 사라지고 inlining 기회가 열린다 (Microsoft Learn: P/Invoke source generation, DEVCLASS, 2022-11-14). Native AOT에서도 이 generator가 필수 — AOT는 런타임 IL emit을 허용하지 않는다.
  • [GeneratedComInterface] (.NET 8+, ComWrappers 기반): [ComImport]가 의존하던 built-in COM marshaller를 대체. 다만 Microsoft가 명시한 한 가지 트레이드오프 — derived interface의 메서드를 호출할 때 [ComImport]는 직접 dispatch가 가능했지만 [GeneratedComInterface]는 base interface로의 cast가 QueryInterface 호출로 변환된다 (dotnet/runtime: ComInterfaceGenerator/DerivedComInterfaces.md). IContextMenu3가 IContextMenu2를 거쳐 IContextMenu를 상속하는 우리 케이스에 직접적으로 부정적이다.
  • CsWinRT: WinRT projection은 자체 ABI 위에서 COM을 호출하며 CsWinRT 2.0부터 trimmable / AOT-compatible (CsWinRT: aot-trimming.md). WinUI 3 자체가 CsWinRT 위에 얹혀 있어 우리가 명시적으로 CsWinRT 코드를 작성하지 않더라도 모든 XAML 콜백이 이 경로를 통과한다.

FileEntry blittability 분석: src/core/file-entry.h:48-59의 40 B 레이아웃을 그대로 P/Invoke로 넘기려면 모든 필드가 blittable이어야 한다.

인터롭 단가 (공개 데이터가 있는 범위):

  • P/Invoke 호출 자체의 절대값에 대한 일관된 마이크로벤치마크는 Microsoft 공식 문서에 수치로 박혀 있지 않다 — 공개 데이터 없음. 방향성만: [LibraryImport] + blittable args → 거의 inline화 가능 (Microsoft Learn: P/Invoke source generation).
  • CsWinRT 2.0의 측정 결과: trimmable projection이 Hello World 기준 디스크 footprint 80% 감소, 시작 시간 7% 개선, WinUI gallery는 디스크 45% 감소 (CsWinRT aot-trimming docs). 이는 per-call 비용이 아니라 트리밍 + AOT 일반 이득.
  • 실측 케이스 (CsWinRT 1373): WinRT 호출 성능이 UWP → WinUI에서 크게 다른 시나리오 — iCloud 사진 폴더(60k 파일) ItemDate indexer가 UWP 14초 vs WinUI 6분 (microsoft/CsWinRT issue #1373). 100k~10m 파일 스캔에서 WinUI 3가 Win32 대비 45-53% 느려진 사례도 보고 (같은 이슈 thread). 측정된 결과이며 우리 hot path와 직접 유사하다.

D.3 100k 항목 관리 객체 비용

이 절이 C# 시나리오의 결정적 변수다. 두 가지 모델을 비교한다.

모델 A: 각 항목을 managed wrapper 클래스로 감싼다

가장 자연스러운 XAML 패턴 — ObservableCollection<FileEntryViewModel>. 문서화된 .NET 객체 헤더 비용: 64-bit에서 모든 reference 타입은 최소 24 B (16 B 헤더 + 8 B 메서드 테이블 포인터, 8 B 정렬) (theburningmonk: Smallest .NET ref type, Simple Talk: Object Overhead).

FileEntryViewModel이 보유해야 할 필드 (현재 FileEntry의 표시용 파생값 포함): name (string), size (long), modified (DateTime 또는 long), attributes (uint), iconIndex (int), groupId (int), displayName cache (string), formattedSize cache (string), formattedType cache (string), formattedDate cache (string).

모델 B: Span<T> / unmanaged backing + WinUI 데이터 가상화

네이티브 메모리(현재 std::unique_ptr<FileEntry[]> + NameArena, file-model-store.cpp:13-14)를 그대로 두고, C# 측은 포인터만 잡은 채 viewport 행에 대해서만 lazy하게 string을 만든다. 이를 위한 WinUI 3의 공식 API:

  • ISupportIncrementalLoading — tail에서 더 가져오는 패턴, 우리의 256-batch 모델과 직접 매칭 가능 (Microsoft Learn: ListView and GridView data virtualization).
  • IItemsRangeInfo — ItemIndexRange 기반의 "viewport에 있는 행 범위" 통지. 우리 모델에서는 viewport 진입 시에만 FileEntry에서 wrapper를 lazy하게 materialize하면 된다 (같은 Microsoft Learn 문서).
  • 주의: IItemsRangeInfo는 instance-당 single ItemsControl 가정. 두 pane이면 두 data source 인스턴스가 필요 (같은 문서).

모델 B에서 Gen2에 적재되는 wrapper는 viewport 크기 × 2 pane × 약간의 lookahead ≈ 수십~수백 개 수준으로 줄어든다. 41 MB가 0.05 MB가 된다. 이게 C# 시나리오를 살리는 유일한 길.

D.4 GC 일시 정지와 완화책

문서화된 GC latency mode (Microsoft Learn: Latency Modes):

  • Workstation GC: 데스크톱 / WPF / WinForms / MAUI 표준 (Nemorize summary). WinUI 3는 기본적으로 workstation GC가 적합.
  • SustainedLowLatency: foreground gen 2를 억제, background gen 2만 허용. 단점 명시: heap 크기가 더 커지고 fragmentation이 증가하며 메모리 압박 시에는 결국 GC가 트리거되어 잠시 멈춘다 (같은 Microsoft Learn). 100k entries 두 pane 환경처럼 heap이 커진 상태에서는 결국 timing 문제로 잠시 멈춤이 발생한다.
  • GC.TryStartNoGCRegion: 사전에 알리는 바이트 양에 한해 GC를 완전히 차단. 짧은 critical section에만 사용 가능, region이 끝나면 이전 mode로 복귀.
  • Workstation GC pause: 측정 사례에서 ~25 ms까지 관찰됨; Server는 ~10 ms (mattwarren.org: Analysing Pause times, 2017 측정값).
  • 현대 데스크톱(60 Hz)에서 frame budget은 16.7 ms — Gen2 pause 하나가 한 프레임을 통째로 먹는다. dotnet/runtime issue 65850에서도 60 fps 유지에 GC pause가 충돌한다는 명시적 보고 (dotnet/runtime #65850).
  • .NET 9의 DATAS (Dynamic Adaptive Tuning for Application Sizing)가 default로 활성화 — long-lived data 크기에 비례해 heap을 조정 (c-sharpcorner: GC in .NET 9). 이게 우리 100k 항목 시나리오에서 heap을 안정시키지만 pause 시간 자체는 여전히 작업 set에 의존.

이 프로젝트에 적용 시 권장 조합 (모델 B를 택한 경우):

  1. Workstation GC + DATAS 기본값.
  2. 스크롤 / 정렬 burst 직전에 SustainedLowLatency 진입, burst 끝난 후 Interactive로 복귀.
  3. 모델 A를 택할 경우엔 stress가 너무 커서 위 mode들이 충분치 않다 — 모델 B가 사실상 필수.

참고: Microsoft 자체도 WinUI 3 / File Explorer 성능 작업("Project K2")의 중심에 GC 압력 감소를 두고 Span<T> + stack allocation 전환을 진행 (Windows News: WinUI 3 Performance Push, Windows News: Project K2). 이 사실 자체가 "기본 패턴으로는 우리 사양에 못 미친다"는 신호.

D.5 Native AOT for WinUI 3 (2026 시점)

Native AOT 지원 현황 (2026-05 기준):

  • WinUI 3 / Windows App SDK 1.6부터 Native AOT가 옵션으로 활성화 (CsWinRT aot-trimming 문서, Windows App SDK 1.6 release notes).
  • Windows App SDK 1.8 stable (2025-09), serviced through 2026-03; SDK 2.0 preview는 .NET 10 타겟 (CTCO: WinUI vs WPF in 2026).
  • AOT 빌드 시 publish는 SelfContained=true가 필수 (dotnet/runtime #109154).
  • 알려진 알파급 결함: win-x86 AOT 빌드에서 GC crash (dotnet/runtime #110607). x86 데스크톱이 우리 타겟에 포함된다면 차단 요인.
  • Microsoft가 보고한 정성적 효과 (File Explorer, Notepad WinUI 3 변환): cold start 40-55% 단축 (Windows Latest, 2026-05-15, Microsoft 텔레메트리라고 출처 표기).
  • AOT가 강제하는 제약: 런타임 코드 생성 금지 — reflection-heavy XAML binding, {Binding}의 일부 시나리오, runtime-generated COM proxy 모두 제거 필요. x:Bind(컴파일 타임 바인딩)와 [GeneratedComInterface]가 권장 경로.
  • 바이너리 크기: UWP 소형 앱 기준 약 +4 MB 증가 (self-contained 때문에) (Microsoft Learn: Modernize UWP with .NET). WinUI 3는 framework + XAML 메타데이터가 더 큼 — 실측 데이터는 앱마다 상이.

이 프로젝트에 대한 함의:

  • AOT를 채택하면 [GeneratedComInterface]가 강제된다 — D.2의 derived-interface QueryInterface 비용이 우리 IContextMenu3 체인에 그대로 부과.
  • AOT 채택 시 [LibraryImport]가 강제 — 좋은 일이지만, 현재 코드에 P/Invoke가 없으니 처음부터 LibraryImport로 작성하면 된다.
  • XAML {Binding} 대신 x:Bind 사용 강제 — 디자이너 productivity 측면에선 큰 차이 없고 오히려 성능에 유리.

D.9 C# 시나리오 핵심 요약

  • 인터롭 경계 빈도는 관리 가능: 진짜 hot path는 LVN_GETDISPINFO 등가 (cell 단위 바인딩) 한 곳. 가상화가 viewport 단위로 제한해주므로 100k 항목 전체가 매 프레임 마샬되지는 않는다. 정렬/그룹/필터 burst만 주의.
  • FileEntry는 그대로 blittable: file-entry.h:48-74의 static_assert 일곱 개가 .NET이 LayoutKind.Sequential blittable 구조체에 요구하는 조건과 일치. P/Invoke로 zero-copy 전달 가능 (이름 문자열만 별도 PtrToStringUni).
  • 관리 객체 wrapper 모델은 금지: 100k 항목을 FileEntryViewModel로 감싸면 약 41 MB 추가 + Gen2 후보 100k개 추가. files-community/Files의 다수 lag 이슈 (#9431, #11082, #16357)와 WinUI 3 / WinRT indexer가 UWP 14초 vs WinUI 6분으로 느려진 사례 (CsWinRT #1373)가 이 경로의 위험을 직접 보여준다. ISupportIncrementalLoading + IItemsRangeInfo를 통해 native-backed lazy materialization만이 현재 C++의 9 MB/pane 메모리 footprint를 유지할 수 있는 길.
  • GC pause는 단일 frame budget(16.7 ms)을 위협: workstation GC pause가 측정 사례에서 ~25 ms (mattwarren.org), dotnet/runtime #65850에서 60 fps와의 충돌이 명시. SustainedLowLatency + 모델 B + .NET 9 DATAS 조합이 필요. 모델 A는 mitigation으로도 부족.
  • Native AOT는 가능하지만 비용도 명확: Windows App SDK 1.6+에서 지원, Microsoft 자체 텔레메트리로 cold start 40-55% 단축 (Windows Latest, 2026-05-15), CsWinRT 2.0 trimmed로 디스크 -45-80%, 시작 -7% (CsWinRT aot-trimming). 그러나 [GeneratedComInterface]의 derived-interface QueryInterface 비용 (dotnet/runtime docs)이 우리 IContextMenu → IContextMenu2 → IContextMenu3 체인에 직접 부과. win-x86 AOT GC crash 회귀 (dotnet/runtime #110607)는 x86 타겟 시 차단 요인.
  • Shell COM 호출은 모두 가능, 단 두 곳이 까다롭다: (a) IDropTarget 위임 패턴 — WinUI 3 element가 자체 RegisterDragDrop을 하지 않으므로 host HWND를 직접 잡거나 managed adapter가 필요; (b) shell context menu owner subclass — AOT에서 callback이 [UnmanagedCallersOnly] static이라 instance state는 GCHandle로 외부 보관. 둘 다 구현 가능하지만 현재 C++ 코드 대비 보일러플레이트 증가.
  • 배포 footprint는 명확히 커진다: framework-dependent도 ~20 MB 수준 (Windows App SDK 사전 설치 필요), AOT self-contained는 더 크다 (SelfContained 강제, UWP +4 MB 기준). "fast-explorer" 포지셔닝에 직접 충돌.
  • 재사용 가능 코드 비율 ~41%: core ~4500 LOC + 정책/포매팅 ~1500 LOC + bench ~1100 LOC ≈ 7100 LOC가 그대로 살아남고, 나머지 ~10800 LOC (특히 main-window.cpp 3242 LOC 전체)는 XAML + ViewModel로 재작성. C++/WinRT 시나리오 대비 productivity 이득은 분명히 있지만 (XAML 디자이너, hot reload, LINQ, async/await), hot-path 위험 영역이 더 많다.

최종 권고 (D 모듈 한정): C# / WinUI 3 시나리오는 모델 B (Native-backed virtualization + ISupportIncrementalLoading/IItemsRangeInfo) + Native AOT + LibraryImport + GeneratedComInterface + Workstation GC + 정렬 burst 시 SustainedLowLatency 가드의 조합에서만 현재 Win32 성능 프로파일에 근접할 수 있다. 이 조합을 못 채우면 (모델 A로 가거나 GC 가드 누락) Files 프로젝트가 겪는 lag 클래스 이슈를 그대로 재현할 가능성이 높다. "C#이라서 자동으로 productivity가 좋아진다"는 진술은 main-window.cpp 3242 LOC 재작성 부담과 동시에 평가해야 한다.

E. 종합 비교 — 예상 성능 수준과 마이그레이션 위험 매트릭스

E.1 핵심 답변 (요약)

"현재 코드는 그대로 두고 UI 도구만 WinUI 3로 교체"라는 시나리오에서, 측정 가능한 모든 신호는 성능이 의미 있게 하향 조정된다는 결론을 가리킨다. 현재 Fast Explorer는 100k 항목 첫 배치 29.83 ms·warm launch 21–36 ms·100k peak working set 9.39 MB로 동급 도메인(파일 탐색기)에서 사실상 최상위권 성능을 측정해 두었고(§A.8), WinUI 3 + Windows App SDK 런타임을 끼우는 순간 framework 자체가 강요하는 비용(빈 윈도우 29–31 ms/frame 리포트, UI 스레드 30 fps cap, AnnotatedScrollBarLabel 100k 할당 10초 회귀)이 그 베이스라인을 깎는다(§B.7). 같은 도메인의 microsoft/Files 가 "cold launch 3.6–5 s vs Explorer 1.55 s", "1–3 s 폴더 네비게이션 지연", "2000-item 폴더 unresponsive"를 자기 이슈 트래커에 보고하고 있다(§B.8) — 동일 backend를 가정하더라도 framework cost는 피하기 어렵다.

예상 도달 범위(추정):

  • Best-case(ItemsRepeater 회피 + ListView·ItemsStackPanel + x:Bind OneTime + ContainerContentChanging·x:Phase + Native AOT + native-backed virtualization + 기존 C++ shell COM 재사용): 현재 대비 start-up 10–20×, scroll latency 2–3×, working set 5–10× 악화 추정. 이론적으로는 "WinUI 3 치고는 매우 빠른" 앱이 된다.
  • Typical-case(Files community 수준 — 비슷한 기능, 검증된 패턴 따랐을 때): cold start 30–150×, folder navigation 30–100×, scroll 2× 이상 또는 회귀. 사용자 체감으로 "현재 Fast Explorer가 주는 즉각성"은 사라진다.
  • 현재 정의된 SLO(§A 표 A.4 design budgets) scroll frame p95 ≤ 16.7 ms, 100k WS target 50 MB / budget 100 MB, warm launch ≤ 500 ms 중 100k WS budget 100 MB와 warm launch 500 ms는 best-case 시나리오에서 여전히 충족 가능하지만, scroll frame 16.7 ms p95는 framework cap(§B.7 #7290)에 의해 위협받는다.

E.4 가장 큰 위험 5가지

  1. UI 스레드 30 fps cap — microsoft-ui-xaml#7290 가 보고하는 "1000 Hz 마우스에도 UI가 30 fps에 묶임"은 현재 Fast Explorer의 scroll frame p95 ≤ 16.7 ms SLO를 직접 위협한다 (§B.7). 회피 패턴이 framework-level 수정에 의존하므로 사용자 코드로 해결 불가.
  2. HWND 4-pane 호스팅 불가 — Microsoft가 인정한 #10050 ("WinUI 3 윈도우 안에 child HWND를 매달면 input이 동작하지 않는다")는 현재 main-window.cpp 3,242 LOC의 페인 호스팅 구조를 강제로 재설계하게 만든다(§C.1).
  3. Shell 컨텍스트 메뉴 owner-draw subclass 동작 미검증 — shell-context-menu.cpp:25-56의 WM_INITMENUPOPUP / WM_DRAWITEM forwarding이 XAML wndProc 환경에서 동작하는지에 대한 문서가 없다(§C.4). 동작하지 않으면 MenuFlyout + 수동 verb 디스패치로 전부 재작성 필요.
  4. ItemsRepeater 100k 항목 회귀 — #8888의 AnnotatedScrollBarLabel 10초+ 할당 버그는 현재 100k 워크로드를 ItemsRepeater로는 운영할 수 없게 만든다(§B.1). ListView+ItemsStackPanel 경로로 가야 하며 ItemsView도 제약 있음.
  5. 아이콘 파이프라인 추가 비용 — HIMAGELIST가 XAML에서 사용 불가, CreateBitmapSourceFromHIcon 부재(discussions#7943). HICON → HBITMAP → SoftwareBitmap → SoftwareBitmapSource 변환층이 매 미스마다 추가되며, 현재 코드의 placeholder pin 패턴과 호환 여부는 별도 설계 필요(§C.5).

E.6 권고 — "현재 성능을 유지하려면"

만약 WinUI 3 마이그레이션을 선택한다면, 현재 성능과 가장 가까운 결과를 얻기 위한 패턴은 다음과 같다(데이터 근거 첨부).

  2. 리스트 컨트롤: ListView + ItemsStackPanel 고정. ItemsRepeater는 100k 회귀(#8888) 회피 위해 사용 금지(§B.1).

  4. Shell COM은 그대로 유지. C++ ShellWorker/IconProvider/DropTarget을 C++/WinRT 윈도우의 child component로 호출(§C.2). 약 700 LOC 무변환.
  5. 아이콘 캐시: HICON→SoftwareBitmapSource 변환 어댑터만 신설. 캐시 키/eviction 정책은 현재 그대로 재사용(§C.5).
  6. Native AOT 적용. WASDK 1.6+에서 cold start 50% 단축(§B.6). 단 win-x86은 GC crash 존재(§D.5)로 x64 only.
  7. 30 fps cap 회피. 현재 시점에 사용자 공간 회피책이 framework 수정 없이는 없음을 사전 수용해야 함(§B.7 #7290).
  8. 4-pane HWND 호스팅 포기. 페인은 XAML UserControl로 재설계, 약 3,000+ LOC 재작성을 비용으로 인정(§C.1).

E.7 결론 한 줄

"UI 도구만 WinUI 3로 교체"라는 가정 자체가 부분적으로 성립하지 않는다 — Shell COM과 core는 살아남지만, main-window/pane HWND 트리·아이콘 파이프라인·컨텍스트 메뉴 hosting은 사실상 재설계 대상이 된다. 그리고 framework 자체가 부여하는 cost(30 fps cap, ItemsRepeater 100k 회귀, WinAppRuntime 베이스 라인) 때문에 가장 신중하게 설계해도 현재 베이스라인(warm 21–36 ms, 100k 첫 배치 29.83 ms, 100k peak WS 9.39 MB)에 도달할 수 없다. 일반적 구현에서는 microsoft/Files 가 자기 이슈 트래커에 적어둔 수준(cold 3.6–5 s, navigation 1–3 s, 2000-item unresponsive)이 현실적 도달점이다.
