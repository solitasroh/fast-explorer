winui_lite — 셸을 갈아 끼울 수 있는 가벼운 UI 라이브러리

메인 윈도우를 쪼개는 김에, 일반 UI 부품을 winui_lite라는 별도 라이브러리로 빼냅니다. 셸에 묶인 부분도 "빈 칸" 형태로 라이브러리에 정의해 두고, 앱이 그 빈 칸을 셸 부품으로 채웁니다. 다른 종류의 데이터(가짜 데이터, 로그, 메모리)로도 같은 UI가 동작합니다.

핵심 아이디어 — "빈 칸"이 뭔가?

라이브러리에는 셸을 직접 부르는 코드는 한 줄도 없습니다. 대신 "누군가 이런 일을 해줘야 함"이라는 빈 칸을 만들어 둡니다.
예를 들어:
  • "이 폴더 안의 아이템 목록을 알려줘" (ItemSource)
  • "이 아이템을 클립보드에 복사해줘" (ClipboardBackend)
  • "이 설정을 어딘가에 저장해줘" (SettingsStore)
  • "이 폴더 안에서 뭐가 바뀌면 알려줘" (ChangeNotifier)
빈 칸 자체는 라이브러리에 있고, 빈 칸을 채우는 실제로 일하는 부품은 앱에 있습니다. 앱은 시작할 때 "이 빈 칸은 이 부품으로 채워주세요" 하고 라이브러리에 알려줍니다.

왜 빈 칸까지 만드나 (그냥 폴더 분리 말고)

  • UI가 셸에서 풀려남. 라이브러리 코드는 "셸"이라는 개념을 모르고도 컴파일됩니다.
  • 가짜 데이터로 UI 테스트 가능. 클릭해서 동작이 맞는지 확인하는 데 진짜 파일이 필요 없습니다 → 자동 테스트가 쉬움.
  • 다른 데이터로 같은 UI 재사용. 나중에 "로그 항목 뷰어"를 만든다면, winui_lite는 그대로 두고 빈 칸을 로그 부품으로 채우면 됩니다. 새 UI 코드는 거의 0줄.
  • 셸 호출이 한 곳에 모임. 지금은 셸 API가 여러 파일에 흩어져 있음 → adapters 폴더 한 곳으로 모임. 셸 관련 버그가 한 곳에서 잡힘.

지금 상태

  • src/ui/main-window.cpp = 3,242줄, 함수 56개.
  • src/ui/ 폴더 = 67개 파일. 절반쯤은 이미 거의 독립적, 나머지는 셸 의존.
  • 셸 API 호출이 약 25개 파일에 흩어져 있음 (clipboard-ops, shell-worker, drop-source/target, address-bar-popup, navigation-cache 등).

꼭 지킬 약속

1. winui_lite는 src/explorer/를 못 봄.
한 방향 의존. lib → src 참조 발견 즉시 빌드 실패하도록 CMake 설정.
2. winui_lite 안에 "셸/탐색기/폴더/파일" 단어 금지.
"item", "location", "value" 같은 일반 용어만 사용. grep 한 줄로 검사.
3. winui_lite는 셸/Shell32/ShellAPI 헤더 include 금지.
Win32 기본 헤더 + 표준 C++만. (objbase.h 같은 COM 기반은 OK)
4. 빈 칸은 작게 유지.
한 빈 칸에 함수 5개 이상이면 두 개로 나눔. 메서드가 1~3개인 게 이상적.
5. 빈 칸 결과는 항상 동기 또는 콜백.
ItemSource가 폴더를 비동기로 읽으면, 결과를 콜백으로 던집니다. winui_lite는 스레드를 만들지 않습니다 — 누가 어떻게 비동기로 처리할지는 채우는 쪽이 정함.

까다로운 부분

1. ListView dispinfo가 핫패스라 가상 함수 호출 비용 우려

스크롤 시 매 행마다 dispinfo가 호출됨. 가상 함수가 너무 비싸면 프레임이 떨어질 수 있음.

해결: ItemDispatcher 빈 칸은 한 번에 여러 행을 받는 batch 함수도 둠. 또는 측정 후 진짜 문제면 그 한 함수만 함수 포인터 + 캐시로 바꿈.

2. 셸 비동기 결과를 콜백으로 받기

셸 폴더 열거는 백그라운드 스레드에서 진행. winui_lite는 스레드를 모름.

해결: 어댑터가 자체 스레드를 만들고, 결과를 PostMessage로 메인 스레드에 던진 다음, 거기서 winui_lite의 콜백 호출. 라이브러리는 스레드 안 만듦.

3. 빈 칸 너무 많아져서 등록 코드가 길어짐

8개 빈 칸 × 등록 줄 → main-window 시작 코드가 길어짐.

해결: winui_lite에 "환경" 묶음 구조를 두고 한 번에 모든 빈 칸을 채워서 전달. 등록은 8줄이 아니라 1줄.

4. 컨텍스트 메뉴는 셸 의존이 깊음

IContextMenu가 직접 그리고 명령도 직접 실행. 빈 칸으로 깔끔히 추상화하기 까다로움.

해결: ContextMenu 빈 칸은 "메뉴 항목 목록 + 선택 시 콜백"만 반환. 셸 메뉴는 어댑터가 자체적으로 처리하고 결과만 보냄.

5. onCommand의 거대 switch

330줄. 명령 라우터로 옮기면 case마다 별도 함수가 됨.

해결: 단계 12에서 menu-actions.cpp에 한 함수씩 작성, 시작 시 라우터에 등록. 한 케이스씩 옮기면서 동작 확인.

이전 두 단계(파일 분리: 6시간 30분, 가벼운 라이브러리: 10~11시간)보다 두 배 가깝게 듭니다. 추가 비용은 빈 칸 설계 + 어댑터 작성 + 가짜 부품. 그 대가로 UI 코드가 셸에서 완전히 풀려납니다.

이번에 안 할 것

  • 사용자 동작 변경 없음. 단축키, 메뉴, 화면, 성능 다 똑같이.
  • 새 기능 추가 없음.
  • 버그 수정 없음 (눈에 띄어도 별도 commit).
  • winui_lite를 별도 git 저장소로 분리 안 함 (당분간 같은 저장소 안).
  • vcpkg/conan 패키지로 만들지 않음.
  • WinUI/XAML 같은 다른 UI 기술로 갈아엎지 않음.
  • 크로스 플랫폼화 안 함 (winui_lite는 Win32 전용 유지).

나중에 가능해지는 것

  • 로그/메모리/DB 뷰어 같은 작은 도구. winui_lite + 새 ItemSource만 작성 → 창/패널/툴바 그대로 재사용.
  • UI 자동 테스트. 가짜 데이터로 시나리오 돌리기 가능.
  • WinUI 마이그레이션이 쉬워짐. chrome 한 층만 갈아 끼우면 됨 (셸 어댑터는 그대로).
  • winui_lite 분리 배포. 안정되면 별도 저장소 / 패키지로.
