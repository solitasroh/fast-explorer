UI 부품을 가벼운 라이브러리로 빼는 계획

메인 윈도우 한 파일(3,242줄)을 쪼개되, 그 김에 "다른 윈도우 앱에서도 그대로 가져다 쓸 수 있는 부품"은 따로 폴더로 빼서 작은 라이브러리로 만듭니다. 파일 탐색기 전용 코드와 분리됩니다.

한눈에 보기

지금:

FastExplorer (실행 파일)
└─ src/ui/  (3,242줄짜리 main-window.cpp + 67개 파일이 한 덩어리)

바꾸려는 모양:

fast_ui (작은 라이브러리, 파일 탐색기 개념을 모름)
└─ lib/ui/  창 만들기, 패널 배치, 스플리터, 툴바, 테마/DPI, COM 도우미...

FastExplorer (실행 파일, fast_ui를 가져다 씀)
└─ src/ui/  주소창, 파일 목록, 클립보드, 폴더 열기 등 탐색기 전용

지금 상태

  • src/ui/main-window.cpp = 3,242줄, 함수 56개.
  • src/ui/ 폴더 = 67개 파일. 일부는 이미 거의 독립적인 부품 (예: com-raii.h, pane-layout.{h,cpp}, splitter-ratios.{h,cpp}, dispinfo-histogram.{h,cpp}) — 사실상 라이브러리 후보.
  • 나머지는 파일 탐색기 전용 (셸 API, 폴더 경로, 파일 복사 등).
  • 지금은 둘이 같은 폴더에 섞여 있어서 어떤 게 재사용 가능한지 눈으로 구분 안 됨.

왜 라이브러리로 빼나

  • 경계가 눈에 보임. "이건 어떤 윈도우 앱이든 쓸 수 있는 부품" / "이건 우리 탐색기 전용" 이 폴더로 갈라집니다.
  • 섞여 들어갈 일이 줄어듬. 라이브러리 쪽 파일에서 "폴더", "파일", "셸" 같은 단어를 못 쓰게 막으면 자연스럽게 깔끔해집니다.
  • 재사용 가능. 나중에 다른 작은 도구 (예: 로그 뷰어, 데이터 비교 툴) 만들 때 이 라이브러리만 가져다 쓰면 창/패널 분할/툴바가 이미 동작합니다.
  • 테스트가 쉬워짐. 라이브러리는 파일 탐색기 없이도 빌드되고 단위 테스트 가능. 의존성이 적으니 빌드도 빠릅니다.
  • 큰 비용 없이 가능. 사실 절반 정도는 이미 독립적입니다. 폴더만 옮기고 빌드 타깃만 추가하면 됩니다.

빌드 모양

지금 CMake 빌드 타깃:

  • FastExplorer (실행 파일)
  • core-tests (테스트)
  • FastExplorerBench (성능 측정)

바뀐 후:

  • 새로 추가 fast_ui (정적 라이브러리, lib/ui 안의 파일들로 만듦)
  • FastExplorer → fast_ui를 가져다 씀
  • core-tests → fast_ui도 같이 테스트
  • 새로 추가 fast_ui_demo (작은 데모 실행 파일, fast_ui만 사용)

꼭 지킬 약속

1. 라이브러리는 앱 코드를 못 봄.
lib/ui 안의 파일이 src/의 무엇이든 #include 하면 안 됩니다. 한 방향으로만 의존합니다 (앱 → 라이브러리).
2. 라이브러리에는 "파일 탐색기" 단어가 안 나옴.
"폴더", "경로", "셸", "파일", "탐색기" 같은 말이 라이브러리 코드/주석/이름에 들어가면 그건 앱에 있어야 할 코드입니다. 검사용 grep 한 줄로 확인 가능합니다.
3. 라이브러리는 Win32 + 표준 C++만 씀.
프로젝트의 다른 어떤 라이브러리도 안 가져옵니다 (단, COM/Shell 헤더는 OK — 그건 OS가 주는 것).
4. 앱은 라이브러리 부품에 "콜백"을 끼워 넣는 식으로 씀.
예: 라이브러리의 "스플리터를 드래그함" 이벤트가 발생하면, 앱이 등록한 함수가 호출되어 패널 재배치를 함. 라이브러리가 앱 함수를 직접 부르지 않습니다.

각 단계마다 매번 하는 일

  1. 옮길/만들 파일 정함
  2. 코드 이동 또는 추출 (몸통은 바꾸지 않음)
  3. include 경로 / CMake 파일 목록 갱신
  4. 빌드 통과 확인
  5. 앱 켜서 해당 부분 동작 확인
  6. commit 1개 (왜 + 무엇)

까다로운 부분

1. 멤버 변수 공유

예전 계획과 같은 문제. 메인 윈도우의 panes_ 같은 데이터를 여러 부품이 봐야 함.

해결: 라이브러리 부품은 "데이터를 들지 않음" 원칙. 필요한 정보는 호출할 때 인자로 받거나, 콜백으로 앱이 넘겨줌. 라이브러리 부품끼리는 데이터 공유 안 함.

2. 메시지 디스패치를 두 층으로 나누는 비용

지금은 handleMessage에서 바로 처리. 라이브러리로 빼면 한 단계 더 거침 → 콜백 호출 비용 살짝 추가.

해결: 핫패스(자주 호출되는 메시지)는 inline 함수 또는 std::function 대신 함수 포인터 사용. 측정 후 실제 차이 있으면 그때 최적화.

3. "거의 일반적인데 살짝 탐색기 색이 묻은" 부품

예: 툴바 row. 일반 그리기 부분과 탐색기 단축키 처리가 한 파일에 섞임.

해결: 단계 6에서 일반 부분만 추출하고, 탐색기 단축키 처리는 앱에 남김. 라이브러리는 "툴바를 그릴 줄 알지만 어떤 버튼을 둘지는 모름" 상태가 됨.

4. onCommand의 거대 switch

330줄짜리 switch가 라이브러리(라우터)와 앱(각 케이스의 실제 동작)으로 갈라짐.

해결: 단계 5에서 라우터 골격만 라이브러리에 둠. 앱이 시작할 때 (메뉴ID, 람다) 쌍을 라우터에 등록. 케이스의 몸통은 앱에 남음.

이전 계획(약 6시간 30분)보다 3~4시간 더 듭니다. 추가 비용은 "라이브러리 경계 정리 + 데모 앱". 그 대가로 재사용 가능 + 경계 명확 + 다음 앱 만들 때 절반 무료.

이번에 안 할 것

  • 사용자 동작 변경 없음. 단축키, 메뉴, 화면, 성능 다 똑같이.
  • 새 기능 추가 없음.
  • 버그 수정 없음 (눈에 띄어도 별도 commit으로 미룸).
  • 라이브러리를 별도 git 저장소로 분리 안 함 (당분간 같은 저장소 안의 폴더).
  • 라이브러리를 다른 빌드 시스템 (vcpkg, conan) 패키지로 만들지 않음.
  • WinUI/XAML 같은 새 UI 기술 도입 안 함.

나중에 (이 계획 후에) 가능해지는 것

  • 두 번째 작은 앱. 예: 로그 파일 비교 뷰어 — fast_ui만 가져다 쓰면 창/4분할/스플리터/툴바가 즉시 동작.
  • fast_ui 별도 저장소로 분리. 안정되면 git submodule이나 패키지로 분리 가능.
  • 각 부품 단위 테스트 강화. 라이브러리 부품은 앱 없이도 테스트 가능 → 빌드/테스트 빠름.
