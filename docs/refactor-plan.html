메인 윈도우 쪼개기 계획

한 파일 3,242줄을 7개 작은 파일로 나누는 작업입니다. 동작은 그대로 두고, 코드 위치만 옮깁니다.

지금 어떤 상태인가

  • src/ui/main-window.cpp 한 파일이 3,242줄입니다.
  • 그 안에 MainWindow 클래스가 가지고 있는 함수가 56개입니다.
  • 이 한 클래스가 지금 다 합니다:
    • 창 만들기, 크기 조절, 다크/라이트 테마 전환
    • 패널 배치 (1패널 / 2패널 좌우 / 2패널 위아래 / 3패널 / 4패널)
    • 주소창 입력 처리, 폴더 열기
    • 메뉴 클릭과 단축키 처리
    • 파일 목록 표시, 정렬, 그룹화, 아이콘
    • 복사 / 잘라내기 / 붙여넣기 / 드래그 / 삭제
    • 상태바 글자 갱신, 선택한 파일 갯수 표시
    • 파일이 바깥에서 바뀌었을 때 새로고침

왜 쪼개려는 건가

  • 새 기능 하나 넣으려고 해도 관련 코드 찾는 데만 한참 걸립니다.
  • 예: "단축키 하나 추가" → 약 3,200줄짜리 함수 묶음 안에서 비슷한 단축키 찾고, 그 옆에 끼워 넣고, 메뉴 ID 정의도 어딘가에서 찾아서 추가해야 합니다.
  • 코드가 한 곳에 다 있으니 한 곳을 고치면 다른 데가 깨질 위험이 큽니다.
  • 지금 이 파일을 여러 명이 동시에 수정하면 깃 충돌이 거의 무조건 납니다.

어떻게 쪼갤 건가

지금 56개 함수를 비슷한 일끼리 7개 묶음으로 나눕니다.

묶음 이름새 파일이 묶음이 책임지는 일
창 본체 main-window.cpp(남는 부분) 창 만들기, 윈도우 메시지 받아서 아래 묶음들에게 전달, DPI 변경, 테마 변경
패널 배치 pane-manager.{h,cpp} 1/2/3/4 패널 만들고 없애기, 패널 크기 계산, 활성 패널 표시
명령 처리 command-router.{h,cpp} 메뉴 클릭과 단축키를 받아서 "지금 무슨 일을 해야 하나" 판단
파일 목록 이벤트 listview-events.{h,cpp} 리스트뷰 클릭/우클릭/정렬/그리기/아이콘 표시
탐색 + 새로고침 pane-navigation.{h,cpp} 주소창 입력 → 폴더 열기, 백그라운드에서 파일 목록 받아서 화면에 채우기, 바깥 변경 감지
클립보드 / 드래그 clipboard-ops.{h,cpp} 복사 / 잘라내기 / 붙여넣기 / 드래그 / 삭제와 그 결과 처리
상태바 status-bar-controller.{h,cpp} 아래쪽 상태바 글자, 선택한 파일 갯수/크기 계산

결과: 메인 윈도우 파일은 약 400~500줄로 줄어듭니다. 나머지는 각 묶음 파일에 들어갑니다. 합산 줄 수는 비슷하지만, 한 파일을 열었을 때 보이는 양이 훨씬 적어집니다.

단계 (한 번에 하나씩)

1준비
  • 메인 윈도우 파일에서 함수들을 위 7개 묶음 중 어디에 속하는지 표시 (주석 한 줄씩)
  • 지금 빌드 통과하고 앱이 정상 동작하는지 확인 (기준선)
2상태바 분리 — 가장 간단
  • 상태바 관련 함수 3개만 새 파일로 옮기기
  • 다른 묶음과 거의 안 얽힘 → 연습 삼아 먼저
3클립보드 / 드래그 분리
  • 복사·붙여넣기·드래그·삭제 함수 묶음 새 파일로
  • 윈도우 메시지 받는 부분만 메인에 남기고, 받자마자 새 파일로 넘김
4파일 목록 이벤트 분리
  • 리스트뷰 알림 처리 함수들 (10개 정도) 새 파일로
  • 가장 큰 묶음 중 하나. 약 700줄 이동 예상
5탐색 + 새로고침 분리
  • 주소창 → 폴더 열기, 백그라운드 결과 처리, 외부 변경 감지
  • 약 500줄 이동 예상
6패널 배치 분리
  • 패널 생성/삭제, 레이아웃 계산, 활성 패널
  • 약 600줄 이동 예상
7명령 처리 분리 — 가장 큰 함수 분해
  • onCommand 함수 하나가 약 330줄 → 큰 switch문
  • case 묶음별로 작은 함수로 쪼개서 새 파일로
  • 마지막에 하는 이유: 다른 묶음이 먼저 빠져나가야 case가 호출할 대상이 명확해짐
8마무리
  • 남은 메인 윈도우 파일 다시 훑어서 안 쓰는 include / 멤버 제거
  • 전체 빌드 + 앱 실행 + 주요 동작 다 눌러보기

각 단계에서 매번 하는 일

  1. 옮길 함수들 목록 만들기
  2. 새 .h / .cpp 파일 만들기
  3. 함수들 그대로 잘라서 새 파일에 붙이기 (몸통은 안 바꿈)
  4. 메인 윈도우에는 "이 묶음에게 일 시키기" 한 줄짜리 호출만 남김
  5. 빌드 통과 확인
  6. 앱 켜서 그 묶음이 책임지는 기능 다 눌러보기 ✓
  7. commit 1개

까다로운 부분

1. 멤버 변수가 다 같이 묶여 있음

예를 들어 panes_ 배열은 거의 모든 묶음이 봐야 합니다.

해결: 일단 메인 윈도우가 데이터 보관소 역할을 합니다. 각 묶음은 메인 윈도우 포인터를 받아서 데이터에 접근합니다. 나중에 (옵션 2 단계에서) 더 깔끔하게 정리할 여지를 남겨둡니다.

2. onCommand가 한 함수에 330줄

메뉴 ID에 따라 분기하는 거대한 switch문입니다. 한 번에 옮기지 말고, 단계 7에서 case 그룹별로 따로 함수로 추출합니다.

3. 윈도우 메시지 라우팅

모든 마우스 클릭, 키보드 입력은 결국 handleMessage 한 곳에 들어옵니다. 이건 메인 윈도우에 그대로 두고, 메시지를 받자마자 해당 묶음에게 넘기는 식으로 합니다.

이번에 안 할 것

  • 사용자가 보는 동작은 하나도 안 바뀝니다. 단축키, 메뉴, 화면 다 똑같이 동작해야 합니다.
  • 새 기능 추가 안 함.
  • 버그 수정 안 함 (눈에 띄어도 별도 commit으로 미룸).
  • 인터페이스 / 추상화 도입 안 함 (그건 다음 단계 옵션 2).
  • 파일 이름 통일 같은 코스메틱 정리 안 함.
