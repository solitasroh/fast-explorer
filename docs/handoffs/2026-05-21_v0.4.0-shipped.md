## What just shipped (v0.4.0)

Published 2026-05-21 at https://github.com/solitasroh/fast-explorer/releases/tag/v0.4.0.

1. **Multi-pane (1..4 simultaneous panes).** `PaneManager` API generalised from dual-only (`openSecond/closeSecond/isDual`) to append/pop (`openPane/closePane`), max 4 slots. All per-slot HWND arrays in `MainWindow` grew from `[2]` to `[4]`. `PaneController` itself is untouched — that's the seam for a future tab-per-pane extension (each "tab" will be one PaneController; the slot becomes a `vector<PaneController>+activeTabIdx`).

2. **10 named layout presets** in `src/core/layout-preset.h`:
   - `Single`, `Dual_V`, `Dual_H`
   - `Tri_A` (left-full + right top/bottom — TC style), `Tri_B` (top-full + bottom left/right), `Tri_C` (3 vertical columns)
   - `Quad_A` (2×2 grid — default), `Quad_B` (4 columns), `Quad_C` (4 rows), `Quad_D` (left-full + right 3-stack)

3. **Grid splitters** (`src/ui/pane-splitter.{h,cpp}`). Registered class `FastExplorer.PaneSplitter`. Up to 3 splitter HWNDs pre-created at `onCreate`, hidden initially, shown/positioned per preset by `relayout`. 1-px hairline visual + 4-5 DIP grab area + XOR ghost-line drag (line follows cursor, single relayout on mouse-up). Ratios per-preset, persisted across restart.

4. **Cycling shortcuts.** Ctrl+1 → Single. Ctrl+2 → Dual_V (or toggle out of dual). Ctrl+3 cycles Tri_A → Tri_B → Tri_C → Tri_A. Ctrl+4 cycles Quad_A → Quad_B → Quad_C → Quad_D → Quad_A. Alt+V / Alt+H keep flipping seam within Dual_V/H only. `lastDualPreset_` remembers V vs H seam choice across Tri/Quad detours.

5. **SessionState v5** (`src/core/settings-store.{h,cpp}`). New fields: `panePaths[4]`, `paneCount`, `activePane`, `preset`, `ratiosPerPreset[10]`. v4 → v5 migrator maps `lastPath` → `panePaths[0]`, `secondPath` → `panePaths[1]`, `layoutMode + orientation` → `preset`. JSON keys: `schema_version: 5`, `pane_paths`, `pane_count`, `active_pane`, `preset` (string), `ratios` (per-preset object).

6. **Status bar shows ACTIVE pane info.** Single full-width part regardless of paneCount; `setPaneStatusText` writes only when `paneIdx == activeIndex()`; `setActivePane` triggers a `refreshSelectionSummary` so the bar updates immediately on focus change.

7. **Delete key works Explorer-style.** Multi-selection + Delete now removes ALL selected items (previously only the focused row). Single-select Delete behaviour unchanged.

Tests: 595/595 passing.

## Post-implementation bug fixes (during user verification)

The 33 plan tasks landed cleanly via subagent batches, but interactive testing surfaced 6 follow-ups before tag. Worth recording because some are non-obvious gotchas:

1. **`b71a021`** — status bar showed only pane 0 in 3/4-pane mode because `kMaxPanes = 2` in `main-window.cpp` gated `setPaneStatusText` and timer ID bands. Single-part design (active-pane drives) plus bump to `kMaxPanes = 4`.

2. **`06b236b`** — multi-select context menu silently failed if any leaf's `ParseDisplayName` returned a stale PIDL. `parseChildren` now skips failures instead of returning empty. Defensive masking.

3. **`ffc9885`** — the BIG one. `isStaleGeneration` was incorrectly dropping 100% of fs-watch + operation-result messages app-wide. Both `fs-watcher.cpp` and `result-channel.h` post with `generation = 0`, but the gate compared against `target->generation()` (non-zero after first openFolder). Regression introduced in Task 30 (`5815f45` original / `28ed1bc` rebased commit message). Symptom: Delete worked at the filesystem level but listview never refreshed, leaving stale OWNERDATA selection bits → multi-select right-click looked broken. Fix: remove the gate entirely; rely on `FsWatcher::stop()` joining the IOCP worker for pre-stop synchronization. Also extended `deleteFocusedItem` to delete the whole selection (Explorer parity) — was only deleting the focused row.

4. **`d377c07`** — splitter drag math was off by `statusH` (~22 px). `axisLengthAtStart` captured at WM_LBUTTONDOWN was full client height, but `computePaneLayout` interprets horizontal ratios against `totalH = clientH - statusH`. Refactor: relayout populates per-splitter `axisOriginInParent` + `axisLengthForRatio` matching the preset math contract. Bonus: `drawGhost` now clips to splitter's logical span (inner splitters in Tri_A right column / Quad_A column halves no longer bleed across the whole window).

5. **`40bf980`** — `restoreLayoutFromSession` ran unconditionally on every launch (even when `loadSessionState` failed), and blindly did `ratiosPerPreset_ = state.ratiosPerPreset`. On first launch / settings.json delete, that wiped the defaults `initRatiosToDefaults()` had just installed, parking every splitter at coordinate 0. Fix: per-preset, skip slots whose three ratios are all zero (keep the defaults).

6. **`4c7115c`** — `ReleaseCapture()` synchronously fires `WM_CAPTURECHANGED` on the same thread. The handler was resetting `ctx->lastGhostPos = -1`. Then WM_LBUTTONUP read `lastGhostPos` AFTER the release → `-1 / clientW → -0.001 → clamped to 0.1 floor` → every drag committed exactly 10% regardless of cursor position. User-visible symptom: "drag does nothing — splitter snaps back". Fix: snapshot all commit-relevant fields into stack locals BEFORE `ReleaseCapture`, pre-clear `dragging` flag so the re-entrant WM_CAPTURECHANGED becomes a no-op. Memory: `feedback_releasecapture_reentrancy`.

## Lessons worth keeping

- **ReleaseCapture is synchronous.** Win32 contract: `ReleaseCapture()` dispatches `WM_CAPTURECHANGED` to the losing-capture window on the same thread before returning. Anything that handler mutates (drag state, ghost line, ...) is visible immediately after the release call. Snapshot fields you need before the release, OR pre-clear the flags the handler keys on so it becomes a clean no-op. Same applies to capture being stolen externally (Alt-Tab, modal dialog, etc.) — WM_CAPTURECHANGED can fire at any time during a drag.

- **`isStaleGeneration` patterns are fragile.** A "drop if generation doesn't match" gate is correct only if BOTH the sender (worker thread) and receiver (UI thread) agree on the current generation. Posting with `gen=0` while the receiver compares against a non-zero target silently drops everything. Prefer pre-stop synchronization (join the worker before bumping the generation) over runtime-gen-check fences when the message volume is bounded.

- **`installPaneAt(idx)` for slots 1..3 must replicate slot 0's setup verbatim** — drop target, fs-watcher, coordinators (icon/selection/label-edit), drag-drop subclass. Missing any step means slots 2/3 silently lack the feature. The plan called this risk out explicitly; the implementation got it right.
