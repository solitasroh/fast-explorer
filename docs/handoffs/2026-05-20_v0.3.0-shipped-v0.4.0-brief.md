## What just shipped (v0.3.0)

Published 2026-05-20 at https://github.com/solitasroh/fast-explorer/releases/tag/v0.3.0.

1. **Address bar: ComboBoxEx → plain Edit + BS_OWNERDRAW `˅` button.** The v0.2.8 → v0.2.12 ship cycle burned five releases trying to dark-theme a ComboBoxEx; the themed Combo renderer owns the focused-state background and ignores every CTLCOLOR / theme-strip / WM_ERASEBKGND attempt. Plain Edit + parent-side CTLCOLOR works.
2. **Vertical centring inside the Edit via WM_NCCALCSIZE subclass** (`addressEditNcPaddingSubclass` in `src/ui/pane-toolbar-row.cpp`). EM_SETRECT/NP are single-line no-ops. The subclass also owns the 1-px border in WM_NCPAINT so removing WS_BORDER + visual-style theme on the Edit doesn't leave it border-less.
3. **Font swap to Segoe UI Variable Display** (Win11 chrome default; auto-falls back to Segoe UI on Win10).
4. **Click on Edit toggles the dropdown popup** — mouse hook in `AddressBarPopup` now excludes the anchor HWND from the auto-hide so `MainWindow::onCommand` can do real toggle.
5. **UNC path support**: `\\server\share\...` enumerates via FindFirstFile on `\\?\UNC\server\share\...`; `\\server` (no share) enumerates the server's shares via WNetEnumResource (mpr.dll). NetShareEnum was tried first but is blocked by modern NAS hosts even when SMB itself works.
6. **Mouse XButton1 / XButton2 → history back / forward** via WM_APPCOMMAND, routed through synthesised WM_COMMAND so the existing nav path runs.
7. Dark-mode polish: column header text (NM_CUSTOMDRAW on listview header), address bar popup TreeView, friendlier network error messages (`ERROR_BAD_NETPATH` → "경로를 찾을 수 없습니다" instead of "내부 오류").

Tests: 558/558 passing.

**Network folder tree expansion is intentionally not implemented** — the underlying Shell API returns 0 items on most configurations (Win Explorer's own Network sidebar has the same problem on this machine). Users navigate UNC by typing `\\server` directly, which works.

## v0.4.0 — 4-pane + grid splitter

User request, verbatim: "분할을 4개까지 확장하자. 거기에 grid splitter 로 크기 조정가능하도록 기능 추가하고".

### Goals

1. Support 1, 2, 3, or 4 simultaneous panes (currently capped at 2).
2. Splitters between panes that the user can drag to resize.
3. Splitter ratios persist across app restart (settings-store).
4. **Performance: 4-pane mode must be as fast as 2-pane mode** — no per-frame layout cost, no extra hot-path branches for the inactive layout slots. User priority is performance.

### Current state to be extended

- `kMaxPanes = 2` somewhere in pane-manager / main-window — needs to grow to 4.
- `PaneManager` exposes `count()`, `at()`, `activeIndex()`, `isDual()`, `openSecond()`, `closeSecond()`. The dual-only API needs generalising.
- Layout: `MainWindow::applyActivePaneAppearance` + `onSize` compute pane rects. Currently switches between single (full window) and dual (50/50 H or V split). For N panes need to support 2×2 grid, 3-pane L/T arrangements, plus 1, 2 H, 2 V.
- Each pane already owns: listview, address bar, dropdown button, toolbar row, pane controller. Just grow the arrays from size 2 to size 4 — most code generalises naturally.
- `std::array<HWND, 2>` instances in `MainWindow`: `listViews_`, `addressBars_`, `addressDropdownBtns_`, `dropTargets_`, `paneToolbarRows_`, `firstBatchSeen_`, `selectionSyncs_`. Bump to 4.
- Layout-switching accelerators in `main.cpp` accel table: Ctrl+1 (single), Ctrl+2 (dual). Add Ctrl+3 / Ctrl+4 for 3-pane / 4-pane.

### Splitter design notes

- Native Win32 doesn't have a splitter control. Two common options:
  - Create a thin child window between panes that captures mouse, draws a 1-px or 2-px line, sets cursor on `WM_SETCURSOR` to `IDC_SIZEWE` (vertical splitter) or `IDC_SIZENS` (horizontal), handles `WM_LBUTTONDOWN` → `SetCapture` → `WM_MOUSEMOVE` → resize neighbours → `WM_LBUTTONUP` → release.
  - Or: hit-test in MainWindow's WM_NCHITTEST / WM_MOUSEMOVE for the gap between listviews, no extra HWND.
- First approach is cleaner; splitter is its own subclass and resize logic is localised. Use a registered class `FastExplorer.PaneSplitter`. Recommend taking this path.
- Splitter thickness: 4 DIP. Hit area can extend a few px each side for easier grab.
- Snap behaviour: optional. Win Explorer doesn't snap.

### Layout modes for N panes

| Panes | Default layout |
|-------|----------------|
| 1     | Full window    |
| 2     | Horizontal split (left/right), splitter is vertical line |
| 3     | Left pane full height + right column split horizontally (top/bottom) — matches Total Commander style. Two splitters: vertical between left and right column, horizontal inside right column. |
| 4     | 2×2 grid. Three splitters: one vertical (between left and right columns) + two horizontal (one per column). Resize the vertical splitter resizes both rows simultaneously. |

User may want different 3-pane / 4-pane arrangements. Ask before committing to the above defaults.

### Persistence

- Add `paneCount`, `splitRatios[]` fields to `SessionState` in `settings-store`.
- Save on app close (existing save path), restore on launch.
- Default ratios: 0.5 for 2-pane, 0.4 vertical + 0.5 horizontal for 3-pane, 0.5 + 0.5 + 0.5 for 4-pane.

### Open design questions to clarify with the user

1. Three-pane arrangement: left full + right column split? Or top full + bottom row split? Or three equal vertical columns?
2. Four-pane: only 2×2 grid, or also support 4-column / 4-row?
3. Splitter visual: hairline + invisible grab area, or visible 4-px bar?
4. Drag feedback: live resize, or ghost line until mouse up?

### Implementation plan (rough)

1. **Brainstorm + spec** with user — answer the 4 open questions above. Use the `superpowers:brainstorming` skill.
2. **Write a plan** — `superpowers:writing-plans` (multi-step, touches PaneManager, MainWindow layout, settings-store, splitter HWND).
3. Bump `kMaxPanes` to 4. Generalise PaneManager API (`openPane(idx)`, `closePane(idx)` instead of `openSecond` / `closeSecond`).
4. Grow per-pane arrays in MainWindow from `[2]` to `[4]`. Add splitter HWND array.
5. Refactor `MainWindow::onSize` to a layout function that takes (paneCount, splitterRatios) and computes pane rects. Single helper, no per-frame branching by pane count.
6. Build splitter control as a registered class with WM_MOUSEMOVE / WM_LBUTTONDOWN / WM_PAINT (just a flat bar with hover tint).
7. Add Ctrl+3 / Ctrl+4 accelerators + (optionally) status-bar layout indicator.
8. Persist + restore from settings-store.
9. Local-build verification (per `feedback_local_test_first` memory) for each layout mode before tagging v0.4.0.
