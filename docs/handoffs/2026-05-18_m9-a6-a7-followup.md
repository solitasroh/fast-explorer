## 학습한 내용 (이번 세션)

### 정책-vs-Win32 분리

- **정책 헬퍼가 가장 작은 테스트 가능 단위**: `chooseSecondPaneInitialPath`는 한 줄 ternary지만 free function + 3 test로 분리해 Win32 dependency 없이 정책 검증. `enterDualMode` 본체는 Win32 dependent라 단위 테스트 어려움 → 정책만 분리하면 unit cover, 통합은 soak runbook으로 cover. "정책을 따로 떼내라"는 원칙이 작은 UX 결정에도 적용됨.
- **fallback capture를 mutation 전으로 이동**: PaneManager.openSecond는 `std::vector<unique_ptr>` push_back으로 raw `pane_` cache 안정성 보장 (unique_ptr 안에 T가 heap), 그러나 그 안정성을 "reader가 reasoning 필요". L2 reviewer 지적 받아 2줄 이동으로 의문 제거. "if it's hard to reason about, restructure rather than comment".

### Schema versioning 운영 패턴

- **lenient(미래 키 무시) + strict(타입 오류 거부)의 동시 운영**: forward + backward compat 동시 달성. `layout_mode = "tri"` (미래 모드) → Single fallback (silent recovery). `layout_mode = 1` (wrong type) → load 실패 (silent corruption 방지). 두 정책이 같은 키에 공존 가능.
- **Writer 헬퍼 추출이 schema 추가의 진입 비용을 낮춤**: 직접 `out.append("\n  \"")` 시퀀스가 7번 반복되던 saveSessionState body가 `appendKey{String,Int,RawString}` 7 라인으로 단순화. 다음 v3 키 추가는 1 라인 edit.
- **`std::string_view` 키 상수**: `kKey*`를 `const char*`에서 `std::string_view`로 바꿔 reader의 모든 key 비교에서 implicit strlen 제거. 호환성 0% 위험, 성능 + 명시성 동시 개선.

### L2 c-cpp-reviewer가 catch한 패턴

- **`noexcept` 거짓말은 silent terminate**: 첫 구현 `std::wstring chooseSecondPaneInitialPath(...) noexcept`. `std::wstring` by value 반환 = copy ctor allocates = `bad_alloc` 가능. `noexcept`면 `std::terminate`. 한 줄 변경 (`const std::wstring&`) 으로 honest, OOM 시 wndProc-level catch로 정상 전파. Pure L1 review가 잡지 못할 가능성, L2 가치 검증.
- **Raw pointer 안정성 reasoning**: `pane_` raw cache가 `paneManager_->openSecond` (vector push_back) 후에도 유효한가? unique_ptr 컨테이너에서는 T*가 stable (T가 heap allocated). 안전하지만 "reader가 매번 reasoning 해야 함" → 코드 순서 변경으로 reasoning 부담 제거.

### 마라톤 운영 (1세션, 2 atom)

- **gap-detector를 close gate로 routine**: M9 본 세션의 학습 그대로 적용. 5/5 invariant 검증 + NOT-READY 0건 확인 후 Report addendum 발행. 만약 gap-detector 없이 Report 갔으면 carry-forward 검증 누락 가능했음.
- **Pre-existing flake 분리 검증**: 5/20 보다 잦은 1/20 flake (`FormatCache_Capacity_EvictsLruEntry`). stash + main HEAD에서 동일 빈도 재현 → 내 atom과 무관 명시 후 commit body에 기록. "내 atom이 회귀시켰는지" 의문을 명확히 분리하는 routine은 가치가 큼.
