1.3.2 WPARAM 패킹 (pane + generation)

멀티 페인 라우팅을 위해 WPARAM 상위 8비트(33-40)에 paneIndex (0..255), 하위 32비트에 generation을 싣는다 (messages.h:108-121): 

inline UINT_PTR makePaneWParam(std::size_t paneIndex,
                               std::uint32_t generation) noexcept {
  return (static_cast<UINT_PTR>(paneIndex & 0xFFu) << 32) |
         static_cast<UINT_PTR>(generation);
}

paneIndex 0이 묵시적으로 디코드되므로 pre-M9 송신자(패킹 helper 미사용)도 pane 0으로 안전하게 해석된다 (messages.h:105-112). MainWindow::isStaleGeneration(main-window.cpp:1215)이 수신 시 generation 검사를 수행해 옛 enumeration 결과를 폐기한다. 단, onOperationResult(main-window.cpp:2472-2500)와 onFsChange(main-window.cpp:2502-2523)는 gen=0로 publish되므로 generation gate를 의도적으로 우회한다 (해당 함수의 주석 참조).

1.5 도입된 알고리즘/이론

  • RAII로 감싼 글로벌 자원: OleScope (main.cpp:62), WinSparkleScope (main.cpp:108), AppServices (app-services.h:18) 모두 wWinMain 스택에 살아 있어 조기 반환 / 예외 / WM_CLOSE 어느 경로로 빠져도 해제 순서가 deterministic하다. AppServices::start가 logger → crash → memory 순으로 부팅하고 stop은 역순으로 내려가 의존성 역추적이 보장된다 (app-services.cpp:16-48).
  • STA(단일 스레드 아파트먼트) 라이프사이클: OleInitialize(STA 진입)와 OleUninitialize의 짝짓기 (main.cpp:64-69) — WinSparkle 백그라운드 스레드 join을 WinSparkleScope dtor가 보장한 다음에야 OleUninitialize가 실행되도록 스코프 중첩이 설계되어 있다 (main.cpp:262-411의 중괄호 블록 참조).
  • 스레드-안전 HWND 게시: g_mainWindowHwnd는 std::atomic<HWND>로 정의되어 WinSparkle 워커가 load(memory_order_acquire)로 읽고 UI 측이 store(release)로 게시 (main.cpp:88, 99, 308, 129). HWND가 사라진 뒤 PostMessage는 잘 정의된(드롭) 동작이므로 추가 동기화가 없다.
  • 예외 차단 wndProc: MainWindow::wndProc이 try { handleMessage } catch (...) { ... }로 감싸 C++ 예외가 Win32 코어로 새지 못하게 한다. WM_CREATE일 경우 -1을 반환해 CreateWindowExW가 실패하도록 강제한다 (main-window.cpp:1416-1428).
  • 제너레이션 기반 stale-message 폐기: makePaneWParam(paneIndex, generation) (messages.h:108) + isStaleGeneration (main-window.cpp:1215)로 옛 enumeration / sort batch를 폐기한다. 다만 ShellWorker 결과와 FsWatcher 통지는 gen=0로 publish되며 의도적으로 gate를 우회한다 (main-window.cpp:2473-2481, 2503-2512의 주석 참조).
  • QPC 기반 디스패치 히스토그램: 메인 루프에서 QueryPerformanceCounter 전후 차이를 1'000'000ULL / hz로 마이크로초 환산하여 StallHistogram에 적재하고, 추가로 classifyStall로 None/Info/Warn/Error 등급 라벨링 (main.cpp:176-189). 동일한 QPC 패턴이 LVN_GETDISPINFO에도 적용되어 DispInfoHistogram에 누적된다 (main-window.cpp:2968-2983).
  • 타이머 밴드를 통한 디바운싱: 3개 타이머 밴드가 페인별로 슬롯을 갖는다 — kTimerFsCoalesceBase=1 (100ms, 파일시스템 변경 합치기), kTimerSelSummaryBase=1+kMaxPanes (80ms, Ctrl+A 폭풍 흡수), kTimerFilterDebounceBase=...+kMaxPanes (80ms, 검색 입력 흡수). 각 페인 idx가 base+idx로 매핑되어 충돌이 없다 (main-window.cpp:56-69, 2277-2303).

1.7.6 분리된 분류(group-by) 메뉴 ID 영역

kMenuGroupByNone=0x8000부터 kMenuGroupByType=0x8003까지 (messages.h:92-95) — 셸 컨텍스트 메뉴의 verb ID 범위 (1~0x7FFF) 바로 위에 배치되어 셸 백그라운드 메뉴에 주입되었을 때 충돌하지 않는다. onCommand 진입부 (main-window.cpp:1953-1979) 가 packed-id 디코드 이전에 이 4개를 먼저 intercept하는데, unpackButton(0x8000)이 잘못된 btn/pane을 만들기 때문이다.

1.7.7 그 외 안전장치

  • MainWindow dtor가 hwnd_이 살아 있으면 DestroyWindow를 호출(main-window.cpp:431-436). hwnd_ 자체는 WM_NCDESTROY에서 nullptr로 클리어된다.
  • WinSparkle shutdown_request_callback은 worker thread에서 호출되므로 g_mainWindowHwnd.load(acquire) 후 PostMessageW(hwnd, WM_CLOSE, 0, 0)만 함 — 문서화된 유일한 thread-safe poke 경로 (main.cpp:98-103).
  • low-memory 콜백은 memory_.setLowMemoryCallback([hwnd]() { PostMessageW(hwnd, kWmFeLowMemory, 0, 0); })로 등록되고 (main-window.cpp:1704-1706), WM_NCDESTROY에서 setLowMemoryCallback(nullptr)으로 해제 (main-window.cpp:1613). ProcessMemoryService 내부가 mutex 아래에서 콜백 사본을 만들고 호출하기 때문에, mid-destruction PostMessage는 잘 정의된(drop) 동작으로 안전하다.
  • WM_NOTIFY가 listView의 children 중 어디서 왔는지를 std::find(listViews_.begin(), listViews_.end(), hdr->hwndFrom)로 검사 (main-window.cpp:2680-2684). 외부 위젯 (스플리터/주소바)의 통지는 별도 경로로 라우팅되므로 listView_ 패밀리 외 통지는 0을 반환한다.

레이아웃의 직렬화 가능한 상태는 두 개의 core enum과 한 개의 ui struct에 모여 있다.

2.4.1 비율 → 픽셀 매핑 (computePaneLayout)

2.4.3 비율 클램핑 정책

두 개의 pure constexpr 함수가 사용된다.

  • computeNewRatio (splitter-resize.h:10): non-cumulative 프리셋용. 기본 [minRatio, maxRatio] = [0.1, 0.9] 로 클램핑하여 0-크기 슬롯이 생기지 않도록 (splitter-resize.h:13-15).
  • computeNewCumulativeRatio (splitter-resize.h:32): Quad_B/C/D 같은 누적 ratio 프리셋용. raw 값은 [0, 1] 로 먼저 클램핑한 다음 이웃 ratio와 minGap = 0.05 의 간격을 유지하도록 [prevRatio + minGap, nextRatio - minGap] 로 다시 클램핑 (splitter-resize.h:42-47). 이로써 컬럼/로우의 inversion을 방지한다고 코멘트가 명시 (splitter-resize.h:28-31).
  • computeRatioFromCursor (pane-splitter.h:49): WM_LBUTTONUP 커밋 경로에서 실제로 호출되는 함수. [0.1, 0.9] 클램핑 하드코딩.

주: 헤더에 computeNewCumulativeRatio 가 정의되어 있으나 본 섹션 파일들 안에서 호출 사이트는 발견되지 않는다 (not present in source within this section's files).

2.4.4 드래그 좌표계 보정

SplitterContext::axisOriginInParent / axisLengthForRatio (pane-splitter.h:28-29) 는 부모 client 좌표에서의 축 원점과 길이를 저장한다. 코멘트가 명시하는 보정:

  • 수직 스플리터: axisOrigin=0, axisLength=clientW (preset 공식은 W * ratio[i]).
  • 수평 스플리터: axisOrigin=reservedTop, axisLength=totalH (= clientH - reservedTop - reservedBottom; preset 공식은 top + totalH * ratio[i]).

코멘트는 부모의 전체 높이를 쓰면 상태표시줄 높이만큼 어긋난다며 이를 v0.4.0의 "drag-feels-off" 버그라고 기록 (pane-splitter.h:23-27). perpLow / perpHigh (pane-splitter.h:35-36) 는 ghost line이 자신이 분할하는 컬럼/로우 밖으로 새어 나가지 않도록 perpendicular 클립 범위를 정의한다.

2.5 정렬과 선택 동기화

2.5.1 정렬 (PaneSortCoordinator)

정렬은 동기/비동기 두 경로로 분기된다. 분기점은 publishedCount() < sortThresholdRows_ (pane-sort-coordinator.cpp:50) 이고 기본 임계값은 kDefaultSortThresholdRows = 2000 (pane-sort-coordinator.h:33) 이다. 코멘트는 "이 미만에서는 sync sort가 UI 50ms 예산 안에 끝나서 jthread spin-up 비용을 낼 가치가 없다"라고 명시 (pane-sort-coordinator.h:32-33).

requestSort (pane-sort-coordinator.cpp:19) 의 의사결정 흐름:

  1. enumerationActive 이면 즉시 Rejected (pane-sort-coordinator.cpp:23-25). 호출자(PaneController)는 이 플래그를 workerActive_.load(memory_order_acquire) 로 전달 (pane-controller.h:99).
  2. 기존 sortWorker_ 가 있으면 stopAndJoin으로 정리하고 pending order를 클리어 (pane-sort-coordinator.cpp:28-36). 코멘트가 "sync sort 후 stale dispatched-sort의 PostMessage가 도착하면 잘못된 permutation 이 적용된다"라는 race를 차단함을 명시 (pane-sort-coordinator.cpp:29-34).
  3. 같은 key로 두 번 호출되면 direction을 토글, 다른 key면 Ascending으로 reset (pane-sort-coordinator.cpp:41-49).
  4. count < sortThresholdRows_: 동기 경로. groupBy == None 이면 store_.sort(spec) 위임, 그렇지 않으면 별도 order 벡터를 만들어 compareWithGroup 로 sort 후 store_.applySortedOrder (pane-sort-coordinator.cpp:50-77). AppliedSync 반환.
  5. 그 이상: sortWorker_ jthread 스폰. order 벡터 채우기 → sort → pendingSortedOrder_ 에 저장 → PostMessageW로 kWmFeSortComplete 전송 (pane-sort-coordinator.cpp:86-110). Dispatched 반환. 두 곳의 tok.stop_requested() 체크 (pane-sort-coordinator.cpp:93, 102) 가 협조적 취소를 보장.

applyPendingSort (pane-sort-coordinator.cpp:114) 는 UI 스레드에서 commit 한다. 주목: request_stop 대신 자연 완료를 기다리는 이유를 코멘트가 명시 — request_stop은 worker가 pendingSortedOrder_를 채우기 전에 끊어 size-mismatch branch 로 빠질 수 있기 때문 (pane-sort-coordinator.cpp:115-117). gen 일치 검사 (pane-sort-coordinator.cpp:121) 와 size 일치 검사 (pane-sort-coordinator.cpp:125) 두 단계로 stale 결과를 폐기한다.

cancel() (pane-sort-coordinator.cpp:134) 은 sorted_ = false로 리셋하지만 sortSpec_ 은 보존해 reapplyAfterEnumeration (pane-sort-coordinator.cpp:143) 이 다음 enumeration 끝난 뒤 사용자 선택을 복원하도록 한다.

2.5.2 선택 동기화 (SelectionSync)

PaneController는 raw entry index 기반의 std::unordered_set<uint32_t> selectedRaws_ (pane-controller.h:286) 를 유지한다. raw index를 쓰는 이유는 "정렬 후 row가 다른 visibleOrder 슬롯으로 이동해도 underlying FileEntry는 같은 raw index를 유지하기 때문"이라고 코멘트가 명시 (pane-controller.h:135-138).

SelectionSync는 두 방향의 동기화를 수행한다.

  • ListView → Pane: handleItemChanged (selection-sync.cpp:40) 또는 handleOdStateChanged (selection-sync.cpp:58) → syncFromListView (selection-sync.cpp:71). per-row delta를 신뢰하지 않고 ListView_GetNextItem(LVNI_SELECTED) 로 전체를 다시 읽어 pane의 selectedRaws_를 재구성한다.
  • Pane → ListView: reapplyFromPane (selection-sync.cpp:105). ScopedFlag (selection-sync.cpp:20) 로 reapplying_를 켜고, 먼저 ListView_SetItemState(-1, 0, LVIS_SELECTED) 로 전체 클리어 후 pane_.selectedRowsUnderCurrentOrder() (pane-controller.cpp:247) 가 반환한 row들에만 LVIS_SELECTED를 다시 set 한다.

전체 리싱크를 선택한 근거는 코멘트에 자세히 적혀 있다: LVS_OWNERDATA list-view가 "deselect-on-click of the previously-selected row" 케이스에서 LVN_ITEMCHANGED / LVN_ODSTATECHANGED 어느 쪽으로도 notification을 안정적으로 보내지 않아서 delta tracking 시 selectedRaws_가 클릭마다 누수되었다 (selection-sync.cpp:75-86). 비용은 O(visible rows) per change, Ctrl+A 100k 행에서도 수 ms라고 같은 코멘트가 평가.

2.6 도입된 알고리즘/이론

  • Pure-function policy 분리: resolveLayoutToggle, nextPresetInCycle, computeNewRatio, computeNewCumulativeRatio, computeRatioFromCursor, defaultRatiosFor, slotCountForPreset, chooseSecondPaneInitialPath 가 모두 constexpr/noexcept free function 또는 inline policy 함수로 격리. pane-layout.h:39-41의 코멘트가 "policy-only (no Win32), constexpr-평가 가능하므로 HWND 없이 단위 테스트 가능 (static_assert)" 이라고 의도를 명시.
  • Thread-aware acquire/release ordering: workerActive_ 는 std::atomic<bool> 로 release-store / acquire-load 페어 (pane-controller.cpp:303, 343) 이며, 코멘트가 "PostMessageW 후의 release-store는 향후 worker-side bookkeeping 도 같은 release boundary 로 보호하기 위함"이라고 설명 (pane-controller.cpp:339-342).
  • Cooperative cancellation (jthread + stop_token): enumeration worker (pane-controller.cpp:307) 와 sort worker (pane-sort-coordinator.cpp:86) 모두 std::jthread + std::stop_token 패턴. jthread-utils.h 의 stopAndJoin helper 사용 (pane-controller.cpp:280, pane-sort-coordinator.cpp:28, 135).
  • Generation 기반 stale-result 폐기: store_.generation() 을 캡처해 makePaneWParam(paneIdx, gen)로 메시지에 실어 보내고, applyPendingSort 에서 gen != pendingSortGen_ || gen != store_.generation() 검사로 navigation 사이에 도착한 결과를 폐기 (pane-sort-coordinator.cpp:121-123).
  • RAII 가드: SelectionSync::ScopedFlag (selection-sync.cpp:20-31) 의 destructor는 std::bad_alloc 에서도 flag를 클리어해야 하므로 noexcept 명시. 코멘트가 "C++ 예외가 wndProc 밖으로 새는 것은 Win32 UB" 라고 근거 기록 (selection-sync.cpp:17-19).
  • State machine for splitter drag: SplitterContext의 dragging 플래그가 idle → dragging → committed/canceled 의 작은 FSM 역할. WM_CAPTURECHANGED 와 WM_LBUTTONUP 가 모두 dragging 해제 경로를 가지며, WM_LBUTTONUP 는 commit 전 모든 필드를 로컬에 스냅샷한 다음 ReleaseCapture를 호출 (pane-splitter.cpp:131-162).
  • XOR rubber-banding: ghost line은 R2_NOTXORPEN 으로 그려져 같은 위치에 두 번 그리면 자기-반전 ("drawing twice at the same position erases", pane-splitter.cpp:12-13) — WM_MOUSEMOVE 마다 previous, new 두 번 호출로 깜빡임 없이 이동.
  • Forwarding container 패턴 (PaneToolbarRow): WM_NOTIFY, WM_DRAWITEM, WM_COMMAND, WM_CTLCOLOR* 가 모두 SendMessageW(parent, …) 로 부모에 위임 (pane-toolbar-row.cpp:863-868, 906-910, 947-950). 자식 윈도의 생성·라이프타임은 MainWindow 가, 위치 조정만 row 가 책임진다 (pane-toolbar-row.h:3-11).

전통적인 GoF observer 명시 구현은 발견되지 않음 (not present in source). 대신 콜백 (SplitterContext::onCommit, pane-splitter.h:15) 과 PostMessage (Win32 message pump) 가 그 역할을 한다.

2.7 주목할 구현 세부사항

2.7.1 DPI 스케일링

PaneToolbarRow 의 모든 픽셀 값은 DIP 상수로 시작해 scaleDip(dip, dpi) = MulDiv(dip, dpi, 96) (pane-toolbar-row.cpp:26-28) 으로 변환한다. 폰트도 DPI별로 재생성 — createIconFont (pane-toolbar-row.cpp:66), createRowFont (pane-toolbar-row.cpp:308). onDpiChanged (pane-toolbar-row.cpp:679) 는 "atomic swap: 먼저 새 폰트 생성 → 자식들에 push → 옛 폰트 delete" 순서를 따른다 ( WM_SETFONT 가 HFONT를 복사하지 않으므로, pane-toolbar-row.cpp:680-683).

스플리터의 두께 (kSplitterThicknessDip) 와 grab half (kSplitterGrabHalfDip) 는 DIP 상수로 정의되어 있으나 (pane-layout.cpp:9-10) computePaneLayout 내부에서는 그대로 픽셀로 쓰인다 — 호출자가 이미 픽셀 단위의 clientWidth, clientHeight 를 넘긴다는 의미이고, DPI 스케일은 호출자가 처리해야 한다 (해당 호출자는 본 섹션 파일들에는 없으므로 not analyzed here).

2.7.2 활성 페인 추적

PaneManager::activeIndex_ (pane-manager.h:65) 가 단일 source of truth. setActive 가 인덱스 범위 검사 (pane-manager.cpp:40-43), closePane 이 activeIndex_ >= panes_.size() 일 때 panes_.size() - 1 로 클램핑 (pane-manager.cpp:34-37). PaneController 측에는 자기 인덱스의 캐시인 paneIndex_ (pane-controller.h:274) 만 있으며, 이 값은 makePaneWParam(paneIdx, gen) 으로 메시지에 실리는 데 쓰인다 (pane-controller.cpp:321, 336).

2.7.3 키보드 포커스 흐름

PaneToolbarRow는 WS_EX_CONTROLPARENT 로 생성된다 (pane-toolbar-row.cpp:357-360). 코멘트가 명시: "IsDialogMessage가 row 안으로 재귀해서 nav 툴바 / 주소창 / 햄버거가 Tab / Shift+Tab 사이클에 들어간다. 이게 없으면 keyboard-only 사용자가 row 에 포커스를 줄 수 없다" (pane-toolbar-row.cpp:354-356). 자식 버튼들은 WS_TABSTOP 을 명시적으로 부여 (pane-toolbar-row.cpp:392, 412).

2.7.4 다크 모드 / 테마

3.3 Drag & Drop 파이프라인

3.3.1 Drop Source

FileDropSource는 자체 데이터를 들지 않는다. QueryContinueDrag의 결정 로직은 단순하다 (drop-source.cpp:26-31):

if (escapePressed) return DRAGDROP_S_CANCEL;
if ((keyState & MK_LBUTTON) == 0) return DRAGDROP_S_DROP;
return S_OK;

GiveFeedback는 항상 DRAGDROP_S_USEDEFAULTCURSORS를 반환하여 shell 기본 커서(↗ copy, ↑ move, ⛔ none)를 그대로 사용한다 (drop-source.cpp:33-35). DoDragDrop을 부르는 측의 코드는 본 모듈 밖에 있어 본 분석 범위에서는 not present in source.

3.3.2 Drop Target — Row vs Background 위임

PaneDropTarget::rebindTarget (drop-target.cpp:60-94)이 파이프라인의 중심이다:

  1. ScreenToClient 후 ListView_HitTest로 커서 아래 행을 탐색 (drop-target.cpp:64-69).
  2. 같은 행이면 리바인딩을 생략. lastHitRow_는 DragOver마다 SHParseDisplayName + SHBindToObject를 반복하지 않도록 캐싱한다 (drop-target.h:50-52, drop-target.cpp:70-72).
  3. 행이 폴더면 currentPath() + leaf로 경로를 합성, 아니면 pane 배경(currentPath())을 사용 (drop-target.cpp:73-85).
  4. queryFolderDropTarget으로 shell IDropTarget을 가져온다 (drop-target.cpp:87).

실제 DROPEFFECT 계산(같은 드라이브 = move, 다른 드라이브 = copy)과 충돌 UI는 shell 폴더의 IDropTarget이 직접 처리한다 — 헤더 주석이 명시한다 (drop-target.h:14-18).

3.3.3 DragEnter/Over/Leave/Drop 흐름

  • DragEnter: currentData_.copyFrom(data)로 데이터 객체를 AddRef하여 보관한 뒤 위임 (drop-target.cpp:96-107).
  • DragOver: 행이 바뀌어 rebound가 true면 이전 target에 DragLeave(clearCurrentTarget이 호출됨)를 보내고 새 target에 다시 DragEnter를 보낸다 (drop-target.cpp:113-115). 이는 shell 폴더가 DragEnter 이후 DragOver만 받는 contract를 따르기 위함이다.
  • DragLeave/Drop: 항상 clearCurrentTarget()로 DragLeave를 forward하고 보관된 데이터 객체를 reset (drop-target.cpp:51-58, 124-141).
  • 현재 target이 없을 때는 *effect = DROPEFFECT_NONE으로 거부 (drop-target.cpp:105, 120, 137).

async drop(IAsyncOperation) 지원은 not present in source.

3.4 클립보드와 Cut/Copy 상태

3.4.1 Copy/Cut 쓰기

ClipboardOps::copy (clipboard-ops.cpp:72-87)는 다음 단계를 수행한다:

  1. bindFolderByPath로 IShellFolder 획득.
  2. buildDataObject가 각 leaf를 ParseDisplayName로 child PIDL로 변환한 뒤 GetUIObjectOf(IID_IDataObject)로 IDataObject를 만든다 (clipboard-ops.cpp:14-39). 하나라도 실패하면 전체 반환을 비움 — 부분 묶음을 허용하지 않는다 (clipboard-ops.cpp:28-31).
  3. stampPreferredDropEffect가 CFSTR_PREFERREDDROPEFFECT를 RegisterClipboardFormatW로 등록하고, GMEM_MOVEABLE HGLOBAL 4바이트에 DROPEFFECT_MOVE(cut) 또는 DROPEFFECT_COPY를 써서 IDataObject::SetData(fRelease=TRUE)로 주입한다 (clipboard-ops.cpp:41-68, 79-80).
  4. OleSetClipboard 직후 OleFlushClipboard를 호출 — "프로세스 종료 후에도 Explorer에서 paste 가능" (clipboard-ops.cpp:84-86).

SetData 실패 시 medium을 호출자가 해제해야 하므로 GlobalFree(mem)으로 직접 회수한다 (clipboard-ops.cpp:62-66) — fRelease 의미를 정확히 따르는 분기.

3.4.2 Paste 읽기

ClipboardOps::paste (clipboard-ops.cpp:89-117)는 IDropTarget으로 위임하는 방식이다:

  1. OleGetClipboard로 IDataObject를 회수.
  2. queryFolderDropTarget(folderPath, ownerHwnd)로 대상 폴더의 shell IDropTarget 획득.
  3. POINTL{0,0}·MK_LBUTTON으로 DragEnter → DragOver → Drop의 합성 시퀀스를 실행 (clipboard-ops.cpp:98-114).
  4. effect가 DROPEFFECT_NONE이면 거부; 성공이면 shell이 자체 진행/충돌 UI를 그린다 — 헤더 주석이 명시 (clipboard-ops.h:25-27).

반환값은 4가지 결과를 구분한다: Success, NoData, NoTarget, Rejected (clipboard-ops.h:11-16).

3.4.3 Cut 시각 표시 추적

CutStateTracker는 "Ctrl+X 표시된 leaf가 어느 폴더에 속하는지"를 단일 폴더 단위로 보유한다 (cut-state-tracker.h:23-25):

  • mark()는 std::move_iterator로 leaf vector를 unordered_set에 흡수 (cut-state-tracker.cpp:5-11).
  • contains(leaf)는 LVIS_CUT 페인팅 결정용 — 헤더 주석에 명시 (cut-state-tracker.h:9-10).
  • empty()는 cutFolderPath_의 비어있음으로 표현 (cut-state-tracker.h:16) — leaves 집합이 아닌 폴더 경로를 sentinel로 사용.

3.5.3 ID 범위와 Extra 서브메뉴 ID 분리

shell verb id 범위는 [kCmdIdMin=1, kCmdIdMax=0x7FFF]로 고정 (shell-context-menu.cpp:12-17). ExtraSubmenu::Item::id는 호출자가 이 범위 위로 부여하도록 강제되며 (shell-context-menu.h:30, 22-26), TrackPopupMenuEx 반환 후 cmd > kCmdIdMax면 shell verb가 아닌 caller의 추가 메뉴 pick으로 분기하여 PostMessageW(ownerHwnd, WM_COMMAND, ...)로 디스패치한다 (shell-context-menu.cpp:261-263). PostMessage인 이유: "TrackPopupMenuEx가 완전히 unwind되고 DestroyMenu가 끝난 뒤에 핸들러가 실행되도록".

3.6 비동기 쉘 작업

3.6.1 ShellWorker 라이프사이클

생성자는 std::jthread를 만들면서 workerMain을 시작한다 (shell-worker.cpp:160-162):

ShellWorker::ShellWorker(HWND host, std::size_t paneIndex)
    : results_(host, kWmFeOperationResult, paneIndex),
      worker_([this](std::stop_token tok) { workerMain(tok); }) {}

소멸자는 request_stop() 후 join()으로 멤버가 사라지기 전에 워커를 회수한다 (shell-worker.cpp:164-173).

3.6.2 STA 초기화와 IFileOperation

워커 스레드는 CoInitializeEx(nullptr, COINIT_APARTMENTTHREADED)로 시작한다 — IFileOperation 요건 (shell-worker.cpp:230-238). 실패 시(예: 호스트가 MTA로 미리 init한 RPC_E_CHANGED_MODE) 루프는 계속 돌며 모든 명령이 silent no-op이 되어 큐는 막히지 않는다 (shell-worker.cpp:232-236 주석).

3.6.3 큐와 stop_token

큐는 std::queue<ShellCommand> + std::mutex + std::condition_variable_any 조합 (shell-worker.h:96-98). dequeueOne (shell-worker.cpp:192-205)은 stop-token-aware wait를 사용한다:

cv_.wait(lk, tok, [this] { return !pendingCommands_.empty(); });
if (tok.stop_requested()) return std::nullopt;

주석: "pending command는 stop 시 폐기 — 호스트 윈도우 종료시점에만 워커가 내려가므로 사용자가 포기한 셈" (shell-worker.cpp:196-200).

3.6.4 IFileOperation 옵션과 verb

silent-recycle 플래그 (shell-worker.cpp:61-65):

FOF_ALLOWUNDO | FOFX_RECYCLEONDELETE |
FOF_NOCONFIRMATION | FOF_NOERRORUI | FOF_SILENT

주석: "FOFX_RECYCLEONDELETE는 휴지통 quota 초과 시에도 휴지통 라우팅 강제 — silent permanent delete로 빠지지 않게" (shell-worker.cpp:56-60). 세 verb는 각각 op->DeleteItem(shell-worker.cpp:110), op->RenameItem(shell-worker.cpp:130), op->NewItem(FILE_ATTRIBUTE_DIRECTORY, folderName, nullptr, nullptr)(shell-worker.cpp:150-151) → PerformOperations.

3.6.5 ComScope 로컬 RAII

shell-worker.cpp의 anon namespace는 독자적인 ComScope<T> 템플릿을 가진다 (shell-worker.cpp:21-54). 이는 com-raii.h의 ComPtr과 별개로, 외부 com-raii.h include 없이 TU 내부 helper로만 동작한다. 외부 ComPtr과 동일한 인터페이스(put(), get(), reset(), bool 변환)를 제공한다.

3.6.6 처리 카운터와 테스트 동기화

processed_는 std::atomic<std::size_t>로 release-store 후 notify_all()하여 (shell-worker.cpp:226-227) 테스트가 std::atomic::wait 기반으로 busy-loop 없이 대기 가능하게 한다 (shell-worker.cpp:183-190).

3.6.7 ResultChannel: 코얼레싱 결과 큐

ResultChannel<T> (result-channel.h:28-78)는 두 가지 핵심 가드를 결합한다:

  • 코얼레싱 PostMessage: postPending_ atomic bool을 compare_exchange_strong로 false→true 전이에 성공한 게시자만 PostMessageW를 1회 보낸다 (result-channel.h:46-54). 누적 배치에 대해 단 한 번의 메시지.
  • swap-and-clear race seal: drainResults가 동일 mutex 임계영역에서 큐 swap과 postPending_ reset을 함께 수행해 "drain 이후 publish 사이 race"가 닫힘 (result-channel.h:57-63).

publish는 rvalue 전용 — "owned handle을 가진 복사 가능 T가 채널과 drain 양쪽으로 silent duplicate되는 것을 막기 위해" (result-channel.h:38-41). 호스트 HWND가 null이면 postPending_을 즉시 false로 복귀 (result-channel.h:51-53).

3.6.8 jthread 정리 유틸

stopAndJoin(jthread-utils.h:11-16)는 "request_stop을 먼저 시그널한 뒤 join" — 주석: "stop 신호 없이 join만 하면 worker가 caller가 무효로 본 state를 계속 통과할 수 있다" (jthread-utils.h:7-10).

3.8 도입된 알고리즘/이론

3.8.1 RAII for COM/Shell handles

  • ComPtr<T>: 복사 금지·이동 허용·소멸 시 Release (com-raii.h:13-25). put()은 reset() 후 T** 반환으로 COM 출력 인자 패턴 지원 (com-raii.h:22). attach vs copyFrom으로 AddRef 의미를 명시 분리 (com-raii.h:27-32).
  • PidlOwner: CoTaskMemFree RAII; release()로 ownership 양도 (com-raii.h:38-62).
  • MenuOwner: DestroyMenu RAII (com-raii.h:64-85).
  • SubclassGuard: SetWindowSubclass/RemoveWindowSubclass 페어 (shell-context-menu.cpp:58-74).
  • ComScope<T>: shell-worker.cpp:21-54의 TU-local helper (ComPtr의 별도 사본).

3.8.2 Reference Counting

구현 COM 객체는 InterlockedIncrement/InterlockedDecrement로 atomic refcount. count==0 시점에 delete this (drop-source.cpp:16-24, drop-target.cpp:41-49).

3.8.3 Apartment Threading

STA(COINIT_APARTMENTTHREADED)는 IFileOperation 워커 스레드에서만 명시 초기화 (shell-worker.cpp:237-238). RPC_E_CHANGED_MODE 실패 시 silent no-op 정책으로 큐 데드락 회피 (shell-worker.cpp:232-236).

3.8.4 Producer-Consumer + Coalesced Notification

ResultChannel의 publish-batch-1-PostMessage 패턴 (result-channel.h:41-55) + drainResults의 swap-and-clear 한 critical section (result-channel.h:57-63). ShellWorker는 std::queue + condition_variable_any의 stop-token-aware wait를 사용 (shell-worker.cpp:193-194).

3.8.5 Hit-Test Caching

PaneDropTarget은 행 hit 결과(lastHitRow_)를 캐싱해 같은 행 위 DragOver마다 발생하는 SHParseDisplayName+SHBindToObject 사슬을 건너뜀 (drop-target.h:50-52, drop-target.cpp:70-72). -2 = uninitialised sentinel.

3.8.6 Outside-Click via WH_MOUSE Thread-Local Hook

AddressBarPopup·SearchPopup 모두 SetWindowsHookExW(WH_MOUSE, ..., GetCurrentThreadId()) + thread_local Self* tOwner 패턴으로 outside click을 감지(각각 address-bar-popup.cpp:304-312, search-popup.cpp:257-262). CallNextHookEx의 hhk를 nullptr로 전달하여 unhook과의 race를 차단 (address-bar-popup.cpp:631-634).

3.8.7 Coalesce-by-Atomic-Flag

postPending_.compare_exchange_strong(false, true, acq_rel) 게이트로 누적 결과에 대해 단 한 번의 PostMessage 보장 (result-channel.h:46-54).

3.8.8 Stop-Token-Aware Wait

std::condition_variable_any::wait의 stop_token 오버로드를 직접 사용 (shell-worker.cpp:193-201). std::jthread의 자동 stop_token + join 의미와 결합 (shell-worker.h:101).

3.8.9 Heap Payload via PostMessage

스레드 경계를 건너 가변 길이 wstring을 보내는 표준 패턴: new std::wstring(...) → PostMessageW의 LPARAM/WPARAM에 포인터, 수신 측 delete. PostMessage 실패 시 sender가 즉시 해제 (address-bar-popup.cpp:573-578, search-popup.cpp:232-238).

3.8.10 fRelease=TRUE 의미 준수

IDataObject::SetData(fe, sm, TRUE)로 ownership을 데이터 객체에 넘기고, 실패 시에만 caller가 GlobalFree (clipboard-ops.cpp:62-67).

4.1 캐시 계층 개요

아이콘 렌더링 경로는 세 계층으로 분리되어 있다.

  • L0 — placeholder HIMAGELIST (IconCache, icon-cache.h:29): 윈도우당 하나의 HIMAGELIST를 소유하고, 폴더/파일 두 개의 synchronous placeholder 아이콘만 미리 채워 둔다 (icon-cache.cpp:34-61). LVS_SHAREIMAGELISTS 스타일이 이 핸들을 참조한다 (icon-cache.h:25-28).
  • L1 — 확장자→슬롯 LRU (ExtensionIconCache, extension-icon-cache.h:28-64): 이미 해결된 시스템 아이콘이 L0의 어느 슬롯에 있는지 확장자 키로 O(1) 평균 조회한다. 기본 256 entry 한계 (extension-icon-cache.h:30).
  • L2 — 백그라운드 STA 워커 (IconProvider, icon-provider.h:34): L1 miss 시 확장자를 큐에 enqueue하고, COM STA로 초기화된 std::jthread 위에서 SHGetFileInfoW를 호출한다 (icon-provider.cpp:95-113). 결과는 ResultChannel로 UI 스레드에 kWmFeIconBatch 메시지로 합쳐 전달된다 (icon-provider.cpp:12-14, result-channel.h:41-55).

이와 별개로 텍스트 컬럼 캐시 (FormatCache, format-cache.h:20)가 존재한다. 이는 아이콘과 무관하게 LVN_GETDISPINFOW 핫패스에서 크기/타입/수정시각 문자열을 키별 LRU로 메모이즈한다. IconCacheCoordinator가 L0~L2를 한 묶음으로 소유하고 (icon-cache-coordinator.h:20-60), FormatCache는 별도로 pane이 보유한다 (소스의 다른 모듈에서 사용).

Placeholder 생성 (createPlaceholderImageList, icon-cache.cpp:34-61): DPI에 맞춰 SM_CXSMICON/SM_CYSMICON을 스케일하고 ImageList_Create(ILC_COLOR32 | ILC_MASK, 2, 8)로 초기 2슬롯, 8슬롯씩 증가하는 list를 만든다 (성장 정책: icon-cache.cpp:29-30의 kInitialImageListSlots=2, kImageListGrowSlots=8). SHGetFileInfoW를 SHGFI_USEFILEATTRIBUTES로 호출해 디스크 I/O 없이 폴더/파일 generic 아이콘을 얻고 (icon-cache.cpp:15-27), 인덱스 0/1에 정확히 들어가는지 검증한다 (icon-cache.cpp:55-59). sentinel 문자열로 L"x"를 쓰는 이유는, 드라이브 루트 형태("C:\\")면 shell이 드라이브 볼륨 아이콘을 반환하기 때문이라고 주석에 명시되어 있다 (icon-cache.cpp:18-21).

Slot 해상도 (IconCacheCoordinator::resolveIconIndex, icon-cache-coordinator.cpp:30-51):

const int placeholder = placeholderIndexFor(entry);
if (isDirectory(entry) || !extensionCache_ || !iconProvider_) {
  return placeholder;
}
const auto extView = extensionView(entry);
if (extView.empty()) return placeholder;
const int cached = extensionCache_->lookup(extView);
if (cached != ExtensionIconCache::kMissingIndex) return cached;
// 첫 사용: placeholder slot에 고정해 두고 워커에 요청

iconProvider_->request(std::wstring(extView));

"첫 사용 시 placeholder 인덱스로 미리 insert" 패턴이 핵심이다. 덕분에 동일 확장자의 후속 행은 placeholder를 그대로 받지만 더 이상 request()를 중복 발사하지 않는다 (icon-cache-coordinator.cpp:47-49 주석 참조).

비동기 로딩 파이프라인 (IconProvider::workerMain, icon-provider.cpp:95-113): 워커는 CoInitializeEx(COINIT_APARTMENTTHREADED)로 STA를 열고, stop-token aware한 cv_.wait(lk, tok, predicate)로 큐를 비운다 (icon-provider.cpp:52-61). 실제 해상도는 resolveIconForExtension (icon-provider.cpp:63-81)이 sentinel 파일명 "x" + extension으로 SHGetFileInfoW를 SHGFI_USEFILEATTRIBUTES와 함께 호출.

Eviction / 메모리 압박 응답 (shrinkIconCache, icon-cache-coordinator.cpp:89-108): kWmFeLowMemory 수신 시 — 새 placeholder-only imagelist를 만들어 iconCache_->swap()으로 교체하고, ListView_SetImageList로 listview 포인터를 다시 걸고, 옛 핸들을 ImageList_Destroy한 후 extensionCache_->clear()로 매핑 테이블도 비운다. "통째 교체" 방식이며 부분 eviction은 없다.

Batch 수신 (onIconBatch, icon-cache-coordinator.cpp:53-87): iconProvider_->drainResults()로 누적된 모든 IconResult를 한 번에 받아, 각 HICON을 ImageList_AddIcon으로 imagelist에 추가하고 슬롯 인덱스를 extensionCache_->insert(ext, slot)로 덮어쓴다. DestroyIcon은 AddIcon 직후 호출 (icon-cache-coordinator.cpp:78-79; Win32 의미: AddIcon이 비트맵을 copy하므로 원본 HICON은 바로 해제 가능). 의존성 누락 시 누수 방지를 위해 results의 HICON을 그대로 DestroyIcon하는 방어 경로가 icon-cache-coordinator.cpp:61-71에 있다.

HICON 누수 방지 종료 시퀀스 (~IconProvider, icon-provider.cpp:16-33): 소멸자에서 worker_.request_stop(); worker_.join();로 먼저 워커를 정지·합류시킨 뒤 results_.drainResults()에 남은 orphan HICON을 DestroyIcon한다. 주석에 "default jthread 파괴는 이 본문 이후에 실행되므로 그 사이 완료된 HICON이 leak된다"는 근거가 명시되어 있다 (icon-provider.cpp:17-22).

4.4 Extension Icon Cache

ExtensionIconCache는 전형적인 list + unordered_map LRU다 (CLRS 11장의 표준 형태). 자료구조는 extension-icon-cache.h:60-63:

std::list<Entry> entries_;
std::unordered_map<std::wstring, EntryIt> index_;
std::size_t capacity_;

키 정규화는 ASCII 핫패스 + 비-ASCII fallback 이중 경로다 (extension-icon-cache.cpp:9-27): 'A'..'Z'는 비트차이 더하기로 빠르게 변환, >= 0x80은 std::towlower로 fallback. 주석은 "Windows 실제 파일 확장자는 사실상 ASCII"라고 가정한다 (extension-icon-cache.cpp:16-19).

lookup(extension-icon-cache.cpp:34-43) 은 hit 시 entries_.splice(begin, entries_, it->second) 로 MRU 위치로 promotion 시킨다. insert (extension-icon-cache.cpp:45-68)는 기존 키면 값 갱신 + splice; 신규면 push_front + 인덱스 등록, capacity 초과 시 entries_.back()을 evict하고 evict된 imagelist index를 evictedOut에 돌려주어 호출자가 슬롯을 회수할 수 있게 한다 (extension-icon-cache.h:42-47).

주의: IconCacheCoordinator의 insert 호출은 두 곳 모두 evictedOut=nullptr로 호출되며 (icon-cache-coordinator.cpp:48,83), L0 imagelist 슬롯은 명시적으로 회수되지 않는다 — 압박 시 shrinkIconCache가 전체를 갈아엎는다.

capacity 기본 256, 0 입력 시 1로 클램프 (format-cache.cpp:13-19). 핵심 helper getOrFill(format-cache.h:75-82)은 hit면 ref 반환, miss면 compute 람다 호출 후 put. 캐시는 UI 스레드 전용이며 워커가 호출하면 안 됨이 명시되어 있다 (format-cache.h:17-19).

4.6 LVN_GETDISPINFO 파이프라인

가상 리스트뷰의 일반적 흐름: LVS_OWNERDATA 리스트뷰는 행 렌더링 직전 각 (row, column)에 대해 LVN_GETDISPINFOW를 보낸다. 본 모듈은 이 콜백 자체의 핸들러 구현은 포함하지 않지만 두 가지 보조 인프라를 제공한다.

(1) Group bridge (ListViewGroupCallback): LVS_OWNERDATA는 기본적으로 LVM_INSERTGROUP과 결합되지 않는다. comctl32의 undocumented IListView::SetOwnerDataCallback (vtable slot 112, listview-group-callback.h:180-182)에 IOwnerDataCallback (IID {44C09D56-8D3B-419D-A462-7B956B105B47}, listview-group-callback.cpp:15-20)을 넘기면 comctl32가 row→group을 이 콜백으로 조회한다.

인터페이스 획득 메커니즘: 헤더는 LVM_QUERYINTERFACE = LVM_FIRST + 189 (listview-group-callback.h:23)와 Windows 7+용 IID_IListView_FE {E5B16AF2-3990-4681-A609-1F060CD14269} (listview-group-callback.h:33, listview-group-callback.cpp:26-31)를 정의한다. Vista의 이전 IID는 더 이상 통하지 않는다는 점이 주석에 명시되어 있다 (listview-group-callback.cpp:24-25). IListView_FE 더미 vtable은 IUnknown(3) + IOleWindow(2) 이후 Slot_05~Slot_111의 stub과 113번째 슬롯 SetOwnerDataCallback으로 정확한 오프셋을 맞춘다 (listview-group-callback.h:61-183). 각 stub에 주석으로 실제 의미(GetImageList 등)를 표시해 Geoff Chappell 리버스 엔지니어링 자료를 추적할 수 있다.

매핑 자료구조 (listview-group-callback.h:239-247):

struct GroupRange { int32_t id; int32_t firstRow; int32_t count; };
std::vector<int32_t> rowToGroupIdx_;  // size == visible row count
std::vector<GroupRange> groups_;       // size == enumeratedIds.size()

rebuild (listview-group-callback.cpp:119-167): 우선 iGroupId → groupIndex 임시 unordered_map을 만든 뒤 visible row를 한 번 훑으며 각 row의 group id를 groupIdForEntry(key, entry, nowFiletime)로 계산해 인덱스로 변환한다. id가 enumerate되지 않았으면 group 0으로 묶어 comctl에 E_INVALIDARG 반환을 피하는 방어 코드가 listview-group-callback.cpp:148-154에 있다. 각 그룹의 firstRow는 첫 행에서만 set, count는 증가시킨다 — 즉 visible order가 이미 그룹별로 클러스터링되어 있다고 전제한다(헤더 주석 listview-group-callback.h:218-222).

  • GetItemPosition/SetItemPosition (listview-group-callback.cpp:58-74): details 뷰만 사용하므로 (0,0) 반환 / no-op.
  • GetItemGroup (listview-group-callback.cpp:92-102): rowToGroupIdx_[itemIndex] O(1) lookup.
  • GetItemInGroup (listview-group-callback.cpp:76-90): groups_[gi].firstRow + itemIndexInGroup O(1).
  • GetItemGroupCount (listview-group-callback.cpp:104-110): 항상 1 — 행 하나가 정확히 한 그룹에 속함.
  • OnCacheHint (listview-group-callback.cpp:112-117): no-op (advisory).

(2) Histogram (DispInfoHistogram): LVN_GETDISPINFO 핸들러에서 측정한 QPC 틱 delta를 누적 (dispinfo-histogram.cpp:26-37: (delta * 1'000'000) / hz). 7개의 upper-exclusive bucket [0,1)/[1,5)/[5,20)/[20,50)/[50,100)/[100,500)/[500,∞) μs로 버킷팅하여 raw sample 저장 없이 p99를 추정한다. 버킷 경계가 50 μs를 포함하는 것은 §14.3 게이트(p99 ≤ 50 μs)에 버킷 경계가 정확히 맞도록 한 의도라고 주석이 밝힌다 (dispinfo-histogram.h:18-21). p99EstimateMicros (dispinfo-histogram.cpp:39-59)는 target = (total * 99 + 99) / 100의 ceil 랭크를 찾아 그 버킷의 upper bound를 반환 (마지막 버킷은 UINT64_MAX). dumpDispInfoHistogram (dispinfo-histogram.cpp:61-98)은 DispInfoHistogramLineSink 콜백으로 헤더 + 7 라인을 내보낸다.

4.7 Filter Pattern

세 모드의 표현 (filter-pattern.h:16-20): Plain(부분 문자열), Wildcard(*, ? 글롭), Regex(std::wregex ECMAScript). 세 모드 모두 ASCII case-insensitive를 std::towlower + C 로케일로 구현 — 비-ASCII는 fold 안 됨이 헤더에 명시되어 있다 (filter-pattern.h:34-37).

구성 시(filter-pattern.cpp:105-111): 쿼리를 toLowerCopy로 소문자화한 뒤 보관. Regex 모드는 tryCompileRegex(filter-pattern.cpp:89-99)가 std::regex_error와 std::bad_alloc을 try/catch로 삼켜 std::nullopt를 반환, 그러면 valid_=false가 되고 matches가 항상 false를 돌려준다. icase 플래그는 명시적으로 제거됨 — 이중 case fold를 피하고 리터럴 fast path를 보존하기 위해 (주석 filter-pattern.cpp:84-88).

매칭 (filter-pattern.cpp:113-142):

  1. empty query → true.
  2. !valid → false.
  3. Regex 모드 + 입력 길이 > kMaxRegexInputWchars=4096 → false (ReDoS 가드, filter-pattern.h:48 / filter-pattern.cpp:117-121).
  4. foldToBuffer로 입력명을 lower-fold. ≤ 256 wchar는 스택 버퍼, 초과는 힙 (kStackBufWchars, filter-pattern.cpp:17, 35-56). 힙 alloc 실패는 oom out-flag로 우회 — matches가 noexcept 계약을 유지하기 위함.
  5. 모드별 분기: Plain은 loweredName.find(query_); Wildcard는 wildcardMatch 호출; Regex는 std::regex_search + try/catch.

Wildcard 알고리즘 (filter-pattern.cpp:62-81): 고전적인 iterative star-backtrack 글롭. starPi/starNi 두 변수로 마지막 * 위치를 기억하고 진행이 막히면 그 시점으로 되돌아간다. 최악 O(name × pat), 평균은 대부분 backtrack 없이 O(name + pat). 패턴 끝의 연속 *를 consume하는 trailing 루프(filter-pattern.cpp:79)가 있다.

모드 자동 감지 (detectFilterMode, filter-pattern.cpp:144-161):

  • "r:" prefix → Regex (prefix 제거).
  • 그 외 * 또는 ? 포함 → Wildcard.
  • 나머지 → Plain.
"r:" 검사가 와일드카드 스캔보다 먼저라서 r:foo.*\.txt 같이 regex 안에 ?/*가 들어가도 정상적으로 regex로 라우팅된다 (주석 filter-pattern.cpp:146-147).

applyFilter (filter-pattern.cpp:163-185): FileModelStore::publishedCount() 전수를 raw index 순으로 훑어 매치하는 항목을 std::vector<uint32_t>로 리턴. empty 쿼리는 0..N-1을 그대로 채운다. assert로 published count가 uint32_t를 넘지 않음을 강제 (4.29B 한계, filter-pattern.cpp:168-169).

IFsBackend(fs-backend.h:74-98)는 두 개의 순수 가상 메서드만 노출합니다:

Result<T>(fs-backend.h:38-56)는 value와 EnumerationError를 함께 담는 단순 sum 타입입니다. EnumerationHandle은 BackendKind tag를 들고 있어(fs-backend.h:65), 각 백엔드의 next()는 handle.kind() != BackendKind::<X>이면 Internal을 반환하는 방식으로 RTTI 비용 없이 핸들 출처를 검증합니다(win32-fs-backend.cpp:355-358, memory-fs-backend.cpp:94-97).

Win32FsBackend

win32-fs-backend.cpp의 핵심은 두 가지 모드를 같은 핸들 타입에 포개는 것입니다(win32-fs-backend.cpp:17-52):

  • Mode::FileFind: FindFirstFileExW 호출 시 FindExInfoBasic + FindExSearchNameMatch + FIND_FIRST_EX_LARGE_FETCH 플래그를 사용합니다 (win32-fs-backend.cpp:324-330). FindExInfoBasic은 short-name(8.3) 채우기를 생략시키며, FIND_FIRST_EX_LARGE_FETCH는 큰 배치로 fetch 하라는 힌트입니다.

5.4 Directory Enumeration

DirectoryEnumerator::run()(directory-enumerator.cpp:55-133)는 self-contained synchronous driver입니다. 클래스 자체는 스레드를 소유하지 않으며, 호출자가 std::jthread로 감싸 사용합니다 (directory-enumerator.h:16-27).

핵심 흐름:

  1. backend.openEnumeration() 시도. 에러가 SharingViolation이고 retry 카운트 잔여가 있으면 interruptibleSleep(config_.sharingViolationRetryMs) 후 다시 시도(directory-enumerator.cpp:65-87). sleep 함수는 stop_token을 kPollingQuantumMs = 10ms 간격으로 폴링해 종료 시 즉시 빠져나옵니다 (directory-enumerator.cpp:37-51).
  2. 루프: backend.next() → batch.push_back(). batch.size() >= config_.batchSize(기본 256, directory-enumerator.h:30)이면 flushBatch() 호출 → store.appendBatch() + onBatch(startIndex, count) 콜백 (directory-enumerator.cpp:20-32, :128-131).
  3. config_.includeHidden == false이면 FILE_ATTRIBUTE_HIDDEN entry를 store 도달 전에 drop (directory-enumerator.cpp:125-127).
  4. 중간에 AccessDenied가 발생해도 store.itemCount() > 0 이면 success로 변환해 부분 결과를 UI에 보냅니다 (directory-enumerator.cpp:106-114).
  5. 취소(stop_token): open/next 경계마다 체크, pending batch는 항상 flush 후 Canceled 반환 (directory-enumerator.cpp:66-68, :97-100).

Store 측 batching은 FileModelStore::appendBatch() (file-model-store.cpp:60-69) → 단건 append를 한 entry씩 수행하며, 한 번 실패(arena/capacity/length)하면 즉시 중단해 실제 저장된 개수만 반환합니다.

5.5 Filesystem Watcher

FsWatcher(fs-watcher.h:18-52, fs-watcher.cpp)는 ReadDirectoryChangesW를 FILE_FLAG_OVERLAPPED로 켜고 CreateIoCompletionPort로 IOCP에 묶어 백그라운드 std::jthread에서 대기합니다(fs-watcher.cpp:22-43).

알림 필터(fs-watcher.cpp:73-75):

constexpr DWORD kNotifyFilter =
    FILE_NOTIFY_CHANGE_FILE_NAME | FILE_NOTIFY_CHANGE_DIR_NAME |
    FILE_NOTIFY_CHANGE_SIZE | FILE_NOTIFY_CHANGE_LAST_WRITE;

버퍼는 alignas(8) BYTE buffer[8192](fs-watcher.cpp:71)이며, 매 iteration 마다 OVERLAPPED을 zero-init 합니다(이전 완료의 Internal/InternalHigh 잔여 방지, fs-watcher.cpp:78-81).

중요한 점: 이 layer에서는 이벤트 파싱·debounce·coalescing이 의도적으로 생략되어 있고 (fs-watcher.h:13-15의 주석에 명시), 단지 bytes > 0이면 "뭔가 바뀌었으니 재평가해라"는 신호로 PostMessageW(target_, message_, wp, 0)를 보냅니다 (fs-watcher.cpp:105-112). 5.7 Name Arena

NameArena(name-arena.h:10-51)는 Windows VirtualAlloc reserve + commit on demand 아레나입니다.

  • 기본 reserve 16MB, commit chunk 64KB (name-arena.h:12-13: kDefaultReserveBytes, kCommitChunkBytes).
  • 생성자에서 reserve를 64KB 배수로 올림 후 VirtualAlloc(MEM_RESERVE, PAGE_NOACCESS) (name-arena.cpp:27-31). NOACCESS로 잡혀 있어 누가 실수로 reserved-but-not-committed 영역을 읽으면 즉시 폴트가 납니다.
  • intern(view)은 size_t 곱셈 오버플로우 가드 (name-arena.cpp:76-83) → 잔여 공간 체크 (:85-91) → 부족하면 한 chunk씩 VirtualAlloc(addr, MEM_COMMIT, PAGE_READWRITE)로 확장 (:93-107) → memcpy 후 view 반환(:108-112).
  • reset()은 VirtualFree(base_, committed_, MEM_DECOMMIT)로 commit만 풀고 reserve는 유지해 즉시 재사용 가능합니다(name-arena.cpp:115-133).
  • 소멸자: VirtualFree(base_, 0, MEM_RELEASE) (name-arena.cpp:33-38).
  • copy 금지, move-only(name-arena.cpp:40-70).

실패 모드: 임의의 intern이 비어 있지 않은 입력에 대해 빈 view를 반환하면 호출자가 fatal-per-pane으로 다루도록 명시되어 있습니다(name-arena.h:27-31).

  • IOCP-driven directory change polling: overlapped ReadDirectoryChangesW + CreateIoCompletionPort + synthetic PostQueuedCompletionStatus wakeup(fs-watcher.cpp:22-43, :51-56).

  • State-based memory notification + throttled re-arm (ProcessMemoryService, process-memory.cpp:180-219).
